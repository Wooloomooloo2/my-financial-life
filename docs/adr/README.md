# Architecture Decision Records

This directory contains the Architecture Decision Records (ADRs) for **My Financial Life**. ADRs capture significant technical and architectural decisions made during the project, including the context that motivated them, the options considered, and the consequences of the decision taken.

---

## What is an ADR?

An Architecture Decision Record is a short document that captures a single architectural decision. Each ADR records:
- **Context** — the situation or problem that required a decision
- **Options considered** — the alternatives evaluated
- **Decision** — what was decided and why
- **Consequences** — the positive outcomes, trade-offs accepted, and ongoing responsibilities

ADRs are written at the time a decision is made and are not retrospective documents. Once accepted, an ADR is not edited to reflect subsequent changes — instead, a new ADR is written that supersedes or amends it.

## Summaries

### ADR-001 — Backend language and triple store
Selects **Python** as the backend language and **Oxigraph** (via `pyoxigraph`) as the embedded triple store, consistent with the sister application My Retirement Life. Oxigraph runs inside the Python process with no separate server, has a very low memory footprint suitable for older consumer hardware, and is fully SPARQL 1.1 compliant. The API layer uses **FastAPI**. Consistency with MRL is an explicit goal — both apps share an ontology and are designed to eventually share a database, making stack alignment a strong reason to adopt the same decisions rather than re-evaluate from scratch.

### ADR-002 — Frontend stack
Selects **HTMX + Tailwind CSS + DaisyUI** as the frontend stack, consistent with My Retirement Life. HTMX enables server-driven UI updates from the FastAPI backend without a JavaScript build pipeline. DaisyUI provides a complete component library including built-in light/dark theming. **Chart.js** (CDN) is used for data visualisation. Consistency with MRL is the primary driver; the stack was already validated for this type of application.

### ADR-003 — Packaging strategy
Defines the distribution approach for non-technical end users: **Windows** (PyInstaller → `.exe`), **macOS** (PyInstaller → `.app` bundle), and **Linux** (PyInstaller → AppImage). Consistent with My Retirement Life. A single build toolchain produces all platform targets. Unsigned macOS builds will show a Gatekeeper warning on first launch until code signing is implemented.

### ADR-004 — Cross-platform portability approach
Defines engineering practices that enforce Windows/Linux/macOS portability: `pathlib.Path` for all file path construction, `platformdirs` for OS-appropriate data directories, `.gitattributes` enforcing LF line endings, `python-dotenv` for configuration, and a prohibition on platform-specific runtime dependencies. Consistent with My Retirement Life ADR-004.

### ADR-005 — Ontology strategy — MRL dependency and MFL extension
My Financial Life reuses the My Retirement Life ontology (`mrl:` namespace) as a shared foundation for currencies, jurisdictions, persons, and the full account hierarchy. MFL-specific concepts (transactions, payees, category rules, import batches, valuation events) are defined in a separate `mfl:` namespace in `mfl-ontology.ttl`. Both TTL files are loaded into a single ontology named graph on startup. Extraction to a neutral shared namespace (`mrl-core`) is deferred until both apps are stable. This decision is recorded in MRL as ADR-010.

### ADR-006 — Instance IRI naming strategy
All user-created instance IRIs follow the pattern **`mfl:ClassName_<uuid>`** where the UUID is generated at creation time (e.g. `mfl:Transaction_3f2a1b`). UUIDs are used rather than incrementing integers (as in MRL) because transactions are created at high volume and UUIDs avoid any risk of collision during bulk import. MRL's integer pattern is retained for low-volume entities shared with MRL (accounts, persons) where human-readability in SPARQL is more valuable.

### ADR-007 — Data access patterns — quad patterns vs SPARQL
Establishes a clear split between the two Oxigraph read mechanisms, consistent with MRL ADR-007: **`quads_for_pattern`** is used for fetching all properties of a known IRI and checking existence. **SPARQL SELECT** is used for filtering, aggregation, multi-hop traversal, and reporting queries. All writes use **SPARQL UPDATE** with explicit XSD datatype annotations on numeric, boolean, and date values.

### ADR-008 — Desktop application UI framework
Replaces the v0.1 browser-based frontend with a native desktop UI built on **PySide6 (Qt for Python)**, packaged as a single Windows `.exe` via PyInstaller. macOS and Linux follow. Qt's Model/View architecture (`QAbstractTableModel` + `QTableView`) is purpose-built for the high-performance, editable, virtualised transaction register the application needs and which HTML/HTMX could not deliver to a Banktivity-grade standard. Tauri, Electron, Flutter, and .NET MAUI were considered and rejected — Tauri because it still renders the UI in a WebView and forces either a Rust rewrite of the import engine or a sidecar process; the others because they discard the existing Python parser/import investment. Supersedes ADR-002.

### ADR-009 — Storage engine for ledger data
Replaces Oxigraph as MFL's primary store with **SQLite**, motivated by the actual workload: tens of thousands of transactions, register filter/sort/search/paginate at sub-100 ms, and per-lot IRR/ROI calculations — all materially easier and faster in SQL than SPARQL. **MRL retains Oxigraph** because its workload (tax law across jurisdictions, heterogeneous evolving relationships) is what RDF was designed for. The "shared database with MRL" goal from ADR-001 moves from the storage layer to the integration boundary — MFL can read MRL's store as a reference source and emit RDF for export, but writes its own data relationally. DuckDB is recorded as a future analytical attachment if reports become a bottleneck. Partially supersedes ADR-001.

### ADR-010 — Transactional schema design
Fixes the concrete SQLite schema implementing ADR-009: nine tables (`person`, `account`, `category`, `payee`, `txn`, `lot`, `valuation`, `rule`, `import_batch`), each carrying both an internal `id` and a cross-app `iri` per ADR-006. **Currency stored as INTEGER minor units (pence)** for exact arithmetic; REAL only for non-currency quantities like share counts. **Categories are hierarchical and dual-sourced** (system/user/import), with `parent_id` self-referencing adjacency and `WITH RECURSIVE` for descendant queries. All system-default categories are deletable except the reserved **Uncategorised** row, which serves as the deletion sink (carve-out enforced at the Repository layer). Import path conflicts auto-create as separate categories rather than prompting or auto-merging. Duplicate detection preserved from v0.1 via a partial unique index on `(account_id, import_hash)`. The reference SQL is in [`docs/schema.sql`](../schema.sql).

### ADR-011 — Account delete policy
The desktop app's account-management UI exposes **hard delete only** for v0.1, with a confirmation dialog that names the account, shows the cascaded transaction count, and warns the action cannot be undone. The schema-level `archived_at` column remains **reserved** for a future Archive UX; `Repository.list_accounts` continues to filter it out so adding archive later is additive (Repository methods + sidebar filter, no schema migration). Matches the user's stated request for this milestone; a non-technical user's mental model of "delete" is preserved; cascading is enforced at the FK layer so deletions are never partial.

### ADR-012 — Payee name-management policy
Defines the policy behind the desktop app's payee-management UI. **Rename collisions reject** with an error that directs the user to Merge — auto-merging on rename would silently reassign hundreds of transactions on what looks like a single-row edit. **Merge is the explicit destructive verb**: the user picks 2+ payees, chooses a target — either one of the selected payees or a brand-new name typed at the picker — and confirms a dialog that names the target and states the reassignment count, with the work done atomically in one SQL transaction. Typing a name that matches an existing payee outside the selection is rejected so a single merge never silently pulls in unselected payees. **Delete preserves transactions and clears their payee** via the schema's `ON DELETE SET NULL` FK rule; the confirmation surfaces this consequence explicitly. Same rename / merge / delete shape is reused for category management (ADR-013).

### ADR-013 — Category management policy
Extends the ADR-012 shape to the hierarchical category tree with the **Reparent** verb added. **Rename collisions reject** per-parent (`UNIQUE(parent_id, name)`); the user is directed to Merge. **Reparent rejects cycles** (target in the moved subtree, via `WITH RECURSIVE` descendant query) and **rejects sibling-name collisions** at the new parent. **Merge rejects sources with subcategories** — the cascade-collision morass of moving children with the merged node is avoided by forcing the user to reparent or merge children first. Merge target may be one of the selected categories or a brand-new top-level name; an existing-outside-selection match rejects per ADR-012. **Delete reassigns transactions to the reserved Uncategorised row** (ADR-010 id=1) after a counted confirmation, **rejects Uncategorised itself**, and **rejects categories with subcategories** rather than silently demoting children via `ON DELETE SET NULL`. Every destructive verb stays at one level at a time; deep restructures are a sequence of small reversible steps, not a single click whose effects are hard to predict.

### ADR-014 — Category kind
Adds a **per-category `kind` column** with values `income`, `expense`, `transfer` (CHECK constraint), so reports and cashflow can interpret signed amounts: a positive amount on an expense category reads as a refund; a negative on an income category as a correction; transfers don't count in either bucket. Migration `0002` adds the column, backfills via recursive CTE (Income subtree → income; everything else → expense), and seeds `Transfer` as the third reserved top-level system root. **New top-level categories pick their kind**; **sub-categories inherit the parent's kind**. **Reparenting across kinds is an explicit confirmation** — confirming cascades the new kind to all descendants. **Merging across kinds is rejected** up front so the user converts kind first via reparent. Uncategorised defaults to `kind='expense'`; reports should surface its transactions as "needs categorising" separately rather than rolling them into the refund total. Import-created top-level categories default to `expense`. Per-row column chosen over derived-from-root because it matches the Banktivity mental model the owner is coming from and avoids forcing a migration of every existing top-level non-system category under a kind root.

### ADR-015 — Account folders in the sidebar
Replaces the flat `QListWidget` sidebar with a two-column `QTreeWidget` (Name | Balance) so accounts can be grouped into Banktivity-style folders. Migration `0003` adds an `account_folder` table (id, name, sort_order, archived_at) and a nullable `account.folder_id` FK with `ON DELETE SET NULL`. Folders are **flat** (no nesting) in v1 — nesting is an additive future migration. Folders are **display-only** — clicking a folder toggles expansion; only accounts and "All transactions" change the register view. Folder-as-view-mode is recorded in the backlog. Balance computation is `opening_balance + SUM(txn.amount)` per account in one query; folder sums are computed in the sidebar, **summed naively across currencies** (documented limitation). Folder verbs: New, Rename, Delete (accounts inside fall to root), Move Up/Down (swap sort_order with neighbour), plus Move Account → Folder ▸ submenu on the account context menu. Account reordering inside folders is deferred per the owner's steer.

### ADR-016 — File save / open model
**No plain Save** menu item — every Repository commit already hits disk, so a "Save" verb would be a no-op and mislead. Two file verbs only: **File → Save Copy As… (Ctrl+Shift+S)** writes an atomic SQLite backup of the current DB to a chosen path via `sqlite3.Connection.backup()` (online + WAL-safe); **File → Open… (Ctrl+O)** closes the current Repository, opens a new one against the chosen file (migration runner upgrades older backups to the current schema), and rebuilds sidebar + category cache + filter combo + register model in one swap. Window title shows the current filename so the user can't accidentally edit the wrong file when juggling backups. File extension: `.mfl` preferred, `.db` accepted on Open; Save Copy As defaults to `.mfl` when the user doesn't type an extension. Document-style dirty tracking is rejected — autosave is universally preferable for financial data, and matching the on-disk-is-truth contract is correct given how Repository writes already work.

### ADR-017 — Bulk edit shape
The register's bulk-edit verb is a **modal dialog with per-field checkboxes**: Payee, Category, Status, Memo. Each row has a leading checkbox that enables its editor; only checked fields are applied to the selection. Empty Payee or Memo with the box ticked **clears** that field on every selected row. Category and Status are NOT NULL, so they have no clear semantics — the user must pick a value. One repository call (`bulk_update_transactions` with an `_UNSET` sentinel that distinguishes "don't change" from "clear to None") runs every checked field's UPDATE inside a single SQL transaction; failure mid-way rolls back the whole batch. Triggers: Transaction menu → Bulk Edit Selected… (**Ctrl+E**) and the table's right-click menu (entry only appears when ≥2 rows are selected). Single-row edits stay on the existing inline editor — the bulk dialog would be noisier for one row. Bulk bar (persistent inline editor strip) and per-field context-menu actions were rejected: bulk bar mixes "what's selected" with "what's editable" in one UI region, and per-field actions break the atomic-across-N-rows feel by requiring one prompt per field.

### ADR-018 — Reports framework + first chart: Spending Over Time
Reports open from a new top-level **Reports** menu into **non-modal `QMainWindow`s**, single-instance per report (re-clicking raises the existing window). Chart library is **QtCharts** (bundled with PySide6 — no extra dep). The first report is **Spending Over Time**: stacked bar chart, granularity = weekly / monthly / quarterly / annually, configurable date range, account filter, category-group filter, Include Uncategorised toggle, dashed average line drawn across the chart, and a summary strip showing total + average + bucket count. Defaults: Monthly, last 12 months, all accounts, all groups, Uncategorised included. Spending semantics: **strict outflow** — `SUM(-amount)` on `kind='expense'` rows where `amount < 0`. Net-spending (refunds reduce) was considered and rejected for v1 because Uncategorised's `kind='expense'` default (ADR-014) misclassifies imported income as refunds, producing negative bar segments that don't render cleanly. Refund handling moves to the future Cash Flow report. **Pie charts are explicitly out** (owner rule). Grouping rule for stack segments: a transaction's category rolls up to its **deepest ancestor whose parent is a root** — so `Expense → Groceries → Tesco` groups as `Groceries`. Uncategorised is its own group with its own toggle. Auto-refresh on every control change (no Refresh button) — at ~1.3k rows the SQL is well under a frame. Average line uses an invisible second `QValueAxis` on the bottom because mixing categorical-x bars and numeric-x lines in one `QChart` needs two x-axes sharing one y-axis.

### ADR-019 — Net Worth report
Three-column non-modal `QMainWindow` modelled on Pocketsmith's Net Worth view, opened from **Reports → Net Worth…**. **Summary** (left): big net-worth total, a horizontal **`ProportionalBar`** showing asset-family widths (no pie — owner rule), colour-coded legend. **Assets** (middle): green header + total, `QTreeWidget` grouped by `account.family` (Cash & Bank, Investments, Property), each group expanded with its individual accounts; "+ Asset" button at the bottom. **Debts** (right): red mirror — liability balances stored negative in the DB are displayed as positive amounts (£429 owed, not -£429); "+ Debt" button. Family → label/colour mapping is a small static table; adding a new family means adding a row there. Investment / property balances in v1 still use `opening_balance + sum(txns)` — the schema's `valuation` table is reserved for mark-to-market but isn't wired yet. Add Account opens the existing `AccountDialog` (same dialog from both + buttons, no preset) and refreshes the register's sidebar after creation.

### ADR-020 — Account transfers
A transfer is **one user intent realised as two linked `txn` rows** — outflow on the source, inflow on the destination — paired by a shared `transfer_id TEXT` column (`mfl:Transfer_<uuid8>` per ADR-006). Migration `0004` adds the column and a partial index. The **verb is the category**: there is no dedicated New Transfer menu item; picking a `kind='transfer'` category on any creation/edit flow (New Transaction, inline cell edit, Bulk Edit) prompts for the destination account and creates the partner row. **`create_transfer`** handles fresh transfers from New Transaction; **`convert_to_transfer`** handles existing-row → transfer-half conversion (inline edit); **`bulk_set_category_and_convert`** handles bulk-edit batches atomically. Direction is encoded in the source row's signed amount — partner gets the opposite sign — so the same conversion logic works regardless of whether the user is entering an outflow or an inflow. **Delete is partner-aware**: `delete_transactions` expands the selection via `expand_transfer_partners` so the user can't strand half a transfer. Payee convention: `create_transfer` sets both halves to "Transfer to/from {account}"; `convert_to_transfer` leaves the source's existing payee alone (preserves import-derived payees) and only the new partner gets the convention payee. Edit-sync, auto un-transfer on category change away from transfer kind, and import-side transfer detection are all deferred to future revisions.

### ADR-021 — Generic CSV column-mapping wizard
Closes the open hole in CSV import: when `_detect_format` returns `"generic"` (Pocketsmith, generic bank CSVs, anything that isn't Banktivity/credit-card/OFX/QFX), a new single-screen modal `CsvMappingDialog` lets the user map Date / Amount / Payee / Memo / Category to columns in their file. **Smart-default pre-fill** scans the file's headers against the alias lists already in `csv_parser.py` (`_DATE_ALIASES` etc., plus a new `_CATEGORY_ALIASES`) so conventional Pocketsmith-style files open with all combos already set — the user just confirms. **Amount style** is a radio toggle between *Single signed column* (with an `Invert sign` checkbox for credit-card-style positive=debit exports) and *Separate debit and credit columns*. **Date format** offers `auto` plus named strptime presets plus a free-text custom. **Live after-mapping preview** re-parses the staged 5 preview rows on every widget change and shows `(unparseable)` cells in red so date-format mismatches and inverted-sign confusions are visible before commit. The dialog is **purely value-producing** — accepts → caller reads `.mapping`, rejects → caller calls the new `ImportService.discard_pending_map(token)` — and `register_window._on_import` now routes the mapped result through the same `_commit_pending()` helper as known-format imports, so the silent-commit-and-status-bar pattern (per the no-dialog-for-known-imports feedback) holds once the format is understood. **No schema change**; no new tables. **Saved mapping profiles are explicitly deferred** — auto-detecting a repeat Pocketsmith CSV by header signature and skipping the dialog is its own future ADR, not bolted onto this one.

### ADR-022 — Register typeahead delegates + inline category create
Replaces the register's bare `QLineEdit`/`QComboBox` cell editors with two repository-backed typeahead delegates so payee and category inline edits match the dialog flows (`make_category_picker`). **`PayeeTypeaheadDelegate`** wraps a `QLineEdit` with a `QCompleter` populated from a new `Repository.list_payee_names()` (contains-match, case-insensitive); new names continue to be silently created via the existing `get_or_create_payee` path. **`CategoryTypeaheadDelegate`** uses the same `make_category_picker` combo as `NewTransactionDialog` and `BulkEditDialog`, reading `list_categories_flat()` fresh on every `createEditor` so a category created inline appears immediately in the next opened editor (no delegate-rebind dance). When the user types a name that doesn't match any category, the delegate calls back to the window which shows a single Yes/No confirm prompt ("Create category 'X' as a new top-level expense category?"); on Yes the category is created with `parent_id=None`, `kind='expense'`, `source='user'` and the cell commit proceeds. Defaults match the import path's `find_or_create_category_path(default_root_kind='expense')` so inline-create and import behaviour are aligned; re-parenting or re-kinding happens in Manage ▸ Categories. A new lightweight `_reload_category_cache()` on the window refreshes the cached choice list and the filter-bar combo without resetting the model, so the inline-create path is safe to call from inside a delegate's `setModelData` (the existing `_refresh_categories_view` still does the full reload + delegate rebind for the import and category-management mutation paths). No schema change.

### ADR-023 — Scheduled transactions
First half of the budget arc — adds the `scheduled_txn` template primitive that the budget round will consume for planned-spending projection. **Schedule-as-template**: one schedule row stores cadence (`weekly` / `biweekly` / `monthly` / `quarterly` / `annual`), an anchor date, a next-due date, an optional end date, an estimated signed amount, a `variable` flag (prompt for actual amount on post), and an `auto_post` flag. Posting materialises exactly one `txn` (or one transfer pair via `create_transfer` if the category is `kind='transfer'`) and advances `next_due_date` — past materialised txns are independent records; the schedule is the template, the txn is the truth. **Manual post by default**, per-schedule auto-post opt-in; a launch-time sweep materialises all `auto_post=1` schedules due through today (loop-based so a user gone two months catches up in one launch). **Anchor-based cadence math** (`min(anchor.day, days_in_target_month)`) so Jan 31 monthly → Feb 28 → Mar 31 (not Mar 28) and Feb 29 annual → Feb 28 in non-leap years. **Transfer schedules reuse `create_transfer`** behaviour from ADR-020 — sign of estimated amount picks source vs destination; `transfer_to_account_id` required when category kind is `transfer`. **`Manage → Schedules…`** single menu entry with table view + New / Edit / Post Now / Delete buttons. **No back-link** from materialised txn to source schedule in v1 (deferred — additive when added). **No `rule` reuse** — categorisation rules and schedule templates are different verbs and stay separate primitives. Migration `0005` adds the table + two partial indexes (one for the auto-post sweep, one broader for the budget projection).

### ADR-058 — Budget redesign (hybrid envelope / zero-sum / long-term planning)
**Supersedes ADR-024 + ADR-025.** Umbrella for a re-design prompted by the owner re-stating the budget's principles after real use of the (always-"rough") original. Twelve principles: multiple budgets + scenarios; a genuine hybrid of envelope + zero-sum + long-term planning (every app picks one); accounts picked first with the **available pool = their balance** (later: credit-card credit as a source + a pay-down feature); a **budget period** (default Jan–Dec); category selection with **historical-average suggestions**; **roll-up budgeting that still tracks children** (budget Bills £500, track electric/gas/council-tax under it); **rollover** (or take the excess elsewhere); a **12-month matrix** of Budget/Actual/Diff by income/expense/transfer; **unbudgeted spending shown as such**; the matrix **editable**; matrix = default annual view, plus a **monthly view** with charts; a **burn-down for any category or the whole budget that projects forward at the current spend rate** (better than Pocketsmith's stop-at-today). The load-bearing pivot: ADR-024's single amortized `amount+cadence` can't support editing one month or rollover, so the model moves to **explicit per-month `budget_allocation` rows** (the matrix is the native object) with a **copy-forward affordance** on edit (just this month / this + later / all 12). Owner-resolved forks: **soft** zero-sum (an `Unallocated = pool − assigned` indicator that reddens but never blocks); **auto-rollover** per line (default on for expenses; `carry_in` computed, not stored; reallocate by editing cells — no move ledger). Migration 0019 extends `budget` (`start_month`/`length_months`/`currency`), adds `budget_line` (envelope: category + role + rollover) + `budget_allocation` (per-month cell), migrates + drops `budget_category`. Actuals + rollover stay **computed** in a rewritten-but-still-pure `budget_calc.compute_matrix`; perimeter cancellation, nearest-budgeted-ancestor bucketing, and paintEvent charts all carry forward. ### ADR-024 — Budget core *(superseded by ADR-058)*
Second half of the budget arc — adds the budget itself: which accounts are in scope (perimeter), which categories have per-period targets at what cadence, and the screen that renders the math. **One budget per file in v1, schema supports many** via dedicated `budget` / `budget_account` (M:N perimeter) / `budget_category` (target + cadence + role) tables (migration 0006). **Perimeter rule**: transfers between two in-perimeter accounts cancel out; transfers crossing the perimeter count as a normal flow. Enforced at SQL via a `NOT EXISTS (… partner …)` subquery in `list_perimeter_txns`. **Per-category amount stored as positive magnitude**, direction inferred from `category.kind` (income/expense/transfer) — single source of truth. **Role on `budget_category` not `category`** (bills / saving / discretionary) — keeps multi-budget future open without category-level lock-in. **Transfer-kind budget categories are skipped in the tile math entirely** (post-ship amendment) — actuals already cancel for intra-perimeter transfers; the planned amount no longer leaks into Income / Bills / Saving / Spending and quietly reduces Available. **Nearest-budgeted-ancestor bucketing** for actuals: a txn on a sub-category lands under the closest ancestor that has a budget; if none, it falls to the "Other" bucket. Matches the user mental model regardless of where they choose to budget (leaves, parents, both). **Pure-Python `mfl_desktop/budget_calc.py`** does the bucketing, pro-rating (`365.25/12` for non-monthly cadences, identity pass-through for matching cadence on its own calendar period), and the four Simplifi-style tiles: Income after bills & saving (planned income − planned bills − planned saving) / Planned spending (sum of discretionary expense budgets pro-rated) / Other spending (actual outflow magnitude on un-budgeted categories) / Available (income tile − planned spending − other spending). **Cash-on-hand badge** in the header separately answers "is my account balance going to survive?" — complementary to the plan-side tile math. **No rollover** in v1. **Two-line non-monthly card display** ships in ADR-025. Setup dialog is a two-tab modal (Accounts + Categories) with Add/Edit/Remove on the category rows; the budget screen is a non-modal `QMainWindow` opened from a new top-level **`Budget → Open Budget…`** menu (`Ctrl+B`), refreshing on `WindowActivate` so flipping back from the register reflects new actuals.

### ADR-028 — Payee aliases, canonical labels, and the auto-categorisation arc (planning)
Planning ADR for a three-round arc that closes the long-running payee-cleanup and rules-engine gaps. Originated from an in-session question about auto-deleting orphan payees after a rename; the conversation broadened to the architecture every mature personal-finance app converges on (Banktivity "Payee Aliases", Quicken "Memorized Payees", YNAB "Payee renaming rules"). **Three concepts kept separate**: canonical payee (preferred label + metadata), alias (raw imported string that maps to a canonical), category rule (independent automation that can fire on either). Owner's two import paths fall out — Path A (fastidious, alias rewrites raw → canonical, category from canonical) and Path B (no cleanup, raw payee stays, category from rule pattern). **Schema choice — Option A self-referential `canonical_id` on the existing `payee` table** (chosen over a two-table split or stuffing aliasing into the `rule` table). One nullable column, one index, no `txn.payee_id` change. `canonical_id IS NULL` means "I am canonical"; non-NULL means "I am an alias of …". Typeaheads filter `WHERE canonical_id IS NULL`; reports roll up via `COALESCE(canonical_id, id)`. Two-level rule (no alias-of-alias) enforced at the Repository layer. **Three rounds in order**: round 1 — data model + manual alias UI (polish-scope, one sitting; closes the "clean my list by hand" case without import-engine changes); round 2 — import-time alias lookup as its own arc (exact match at minimum, pattern / fuzzy as follow-ups); round 3 — rules engine as its own arc (long-deferred CLAUDE_CONTEXT backlog item, finally wired up). Recommended scheduling: round 1 before Reports round 2 (typeahead and rollup helpers make the hierarchical category picker cleaner); rounds 2 and 3 after Reports round 2. **ADR-012 will be amended after round 1** to record the canonical/alias model and the "merge vs alias" distinction. **CLAUDE_CONTEXT "Categorisation rules engine" backlog item absorbed into round 3.** **Orphan-cleanup question dissolved** — typos become aliases (cheap, recoverable), not deletes (lossy). No round-1 implementation in this ADR; per-round implementation ADRs follow when each round starts.

### ADR-035 — Multi-currency foundation
Bakes true multi-currency support into the data layer before the first USD account is imported. Migration 0009 adds three tables — `setting` (flat key/value: openexchangerates API key, last-refresh timestamp, matcher tunables, future prefs), `fx_rate(date, base, quote, rate, source)` (daily granularity; provider rows are USD-base per free-tier constraint; manual entries allowed in any direction), and `transfer(iri, from_account_id, to_account_id, rate, rate_source)` (parent row per pair; matches `txn.transfer_id` IRI; `rate_source` ∈ {derived, manual, fx_rate}). Existing same-currency pairs backfilled at rate=1.0. **`txn` does NOT gain a currency column** — a txn's currency is its account's currency, single source of truth; `account.currency` and `person.base_currency` already existed in 0001. **`Repository.get_fx_rate_nearest` six-step lookup**: exact bilateral → exact inverse → exact USD-pivot → nearest-prior bilateral → nearest-prior inverse → nearest-prior USD-pivot. Inverse step is critical — without it, asking for `GBP→USD` against a USD-base feed silently returns no rate. `convert_amount` is the single conversion path; same-currency early-exits. **Transfer plumbing is end-to-end currency-aware**: `create_transfer`, `convert_to_transfer`, `post_scheduled_txn` each take optional `to_amount` / `rate` / `rate_source` with a two-of-three resolution rule (supply any one and the others derive; supply none and FX-table is consulted with nearest-prior fallback). Destination row's stored amount is the receiving-side magnitude, not a converted echo of the source. `transfer.rate` records intent; the two txn amounts are the truth-of-money. **`mfl_desktop/fx.py`** is a no-Qt urllib client + `refresh_latest_into(repo, force=False)` (skips when last refresh <24h ago and when no non-USD accounts exist — single-currency users pay no API budget) + `backfill_historical()` for explicit ranges. **Background launch refresh** wired in `__main__.py` via `QThreadPool` with its own Repository connection. **Manage → Currencies… dialog** holds API key (with "stored in this file" disclaimer), Refresh Now, latest-rates table, Add manual rate row, and the ADR-036 matcher tunables (window-days default 3, FX tolerance default 1.0%). **Rejected**: per-txn currency column (deferred for the rare manual foreign-currency-on-domestic-card case), per-budget currency override (deferred), OS-keychain key storage (acceptable v1, follow-up ADR if real sharing becomes a thing). **Deferred follow-ups**: report display-currency selector + conversion (the headline reason for the arc); `backfill_historical` wired into the Currencies dialog with a cost-confirmation per the ADR's guard-rails; provider rotation hooks.

### ADR-036 — Transfer matching: link existing other-side row instead of always creating a partner
Replaces ADR-020's always-create-partner behaviour with a matcher that finds an unmatched candidate on the destination account first. Critical for the import-everything-then-categorise workflow — without matching, importing both sides of a transfer separately and then marking one as transfer creates a duplicate inflow on the destination. **`Repository.find_transfer_candidates`** filters by opposite-sign amount, `transfer_id IS NULL`, posted_date within ±`transfer_match_window_days` (default 3), and (for cross-currency) amount within `transfer_fx_tolerance_pct × 5` (default 1.0 × 5 = 5%) of `source_magnitude × spot_rate`. Returns scored candidates ranked best-first. **`Repository.link_transfer`** writes a fresh `transfer_id` IRI on both rows, **rewrites both rows' `category_id`** to the source's transfer category (the rewrite is the whole point — the user's mental model is "this inflow *is* the other half"), and inserts a `transfer` parent row with the rate back-derived from the two amounts. **`mfl_desktop/transfer_reconcile.py`** holds pure-Python `score_candidate` (weights: 100 base, −5×|days_apart|, −50×amount_mismatch_pct/100, −20 if currencies differ, +10 on payee-token overlap with a stopword filter that skips "transfer/from/to/payment/deposit"), `strength_for_score` bins (≥80 Strong, ≥60 Good, else Possible), and a generic `greedy_pair` helper used by ADR-037. **UI dialogs** in `mfl_desktop/ui/transfer_match_dialogs.py`: `TransferMatchConfirmDialog` (one candidate; [Match] / [Create new partner] / [Cancel]), `TransferMatchPickerDialog` (many candidates; ranked table with strength chips + trailing "Create new" sentinel row), and `BulkTransferReviewDialog` (per-row decision cell — fixed label for unambiguous rows, inline combo with top-score pre-selected for ambiguous ones). All wired into `register_window.py`: the inline category-edit flow routes through `_offer_transfer_match` before falling back to `convert_to_transfer`; the bulk-edit dispatcher does phase-1 field updates (category + payee/status/memo) then runs the matcher per row, opens the review dialog, and commits via `bulk_match_or_create_transfers` (single SQL transaction; mixed LinkExisting / CreateNew plan). Matcher errors fail soft — status-bar message + fallback to create-new, so a missing FX rate never blocks the flow. **Rejected**: auto-decide on one strong candidate (owner explicitly asked for confirmation — silent linking is too easy to get wrong); a Hungarian-algorithm optimal bipartite matcher in the reconcile path (greedy is good-enough for personal-finance shapes and ~30 lines of pure Python); per-account-override of the window/tolerance (acceptable v1; per-bank-settle-latency follow-up if real use surfaces it).

### ADR-051 — Split transactions
Makes split transactions first-class — one payee/date/total divided across several categories (each with its own memo and a **signed** amount, so a −120 groceries line + a +20 cashback line net to −100). Until now the Banktivity CSV importer flattened splits into the memo and filed them under Uncategorised, mis-attributing them in every report/budget. **Key decision: a new `txn_split` child table holds the lines while the parent `txn` keeps the full signed total**, so the entire money layer (balance, `balance_as_of`, `statement_residual`, holdings, net worth, running balance, dedup hash) is **untouched** — only *category attribution* unrolls. Child-`txn` rows and JSON-on-txn were rejected (they'd force every `SUM(amount)` query to de-dup, or aren't queryable). A single SQL **view `txn_category_line`** (non-split txns map to themselves; splits explode into one row per line) serves the two SQL category consumers with a one-line `FROM` swap (`spending_aggregates`, `list_perimeter_txns` — its transfer-cancellation subquery stays on base `txn`); the per-account Top-Categories screen unrolls in memory via `split_lines_for_txns`. Migration 0017 adds the table (FK mirrors `txn.category_id` — `NOT NULL`, no `ON DELETE`; the category delete/merge path repoints split lines to the sink) + the view; the invariant `Σ lines == parent total` is asserted on every write (integer pence, exact). In the register a split shows as **one row reading "—Split—"** and is **fully dialog-edited** (non-editable inline, like investment rows); double-click opens the new **`SplitTransactionDialog`** (header + a category/memo/signed-amount line table + a live "Unassigned" gate that must hit 0; transfer-kind categories excluded — split ⊥ transfer in v1). A **"Split…"** button on New Transaction routes the header+amount into the same dialog; **Bulk Edit** prompts before converting split parents to a single category. Import: the parser→service dict gains an optional `splits` key (Banktivity-only); `_collapse_banktivity_splits` now emits structured lines instead of memo text, **auto-balancing** with an Uncategorised remainder line when the source doesn't sum (never rejects, no float tolerance); the dedup hash stays parent-level. **Scope (v1): cash/bank/credit only** (investment rows keep the ADR-048 dialog); QIF split import (`S`/`E`/`$` in bank sections) is a deferred extension of the same `splits` contract. **Verified headless on a copy of the live DB:** split insert holds the invariant and surfaces `split_count`; converting plain↔split leaves balances and statement residual byte-identical; spending unrolls to line categories; a Banktivity split block imports as parent + lines, an unbalanced block gets a remainder line, and re-import dedups to zero. (Qt-runtime smoke of the model flags/dialog awaits the project venv — PySide6 isn't installed on the dev Mac yet.)

### ADR-052 — Merge securities
Fills the long-standing "merge-securities verb is still unbuilt" gap (open since ADR-044). Because `security.name` is `UNIQUE` and the importer keys on it, a renamed fund lands as **two records sharing one ticker** — the owner's TSBIX split into "Nuveen Core Impact Bond R6" (id 47, ticker, 219 Tiingo prices, 23 txns) and "TIAA CREF CORE IMPACT BD INST" (id 65, 131 txns, 59 transaction prices), so holdings and the Returns report saw two half-positions. ### ADR-053 — In-kind share transfers (ShrsIn / ShrsOut) are not sales
Fixes a phantom realized loss: the owner moved a position Morgan Stanley → E\*Trade as an in-kind transfer (`ShrsOut` from one account, `ShrsIn` into the other, $0 cash), and the Returns report showed TSBIX with a **−$52,865 realized loss** that never happened. Cause: `shrsout` sat in `SHARE_OUT_ACTIONS` with `sell`, so it was booked as a disposal with $0 proceeds (`realized = 0 − full cost basis`), while the matching `ShrsIn` re-added the shares at zero basis. Fix: a new `SHARE_TRANSFER_ACTIONS`/`is_share_transfer` (in `qif_actions.py`) marks ShrsIn/ShrsOut as custodian moves; `holdings.py` (all three engines — `compute_holdings_view`, `compute_returns`, `compute_value_history`) now removes shares on a `ShrsOut` with **zero realized gain** and parks the popped FIFO lots in a per-security **transfer pen**, from which the matching `ShrsIn` pulls its **carried cost basis** — so a clean out→in transfer nets to no realized impact and preserves basis. The Returns report (`investment_returns_window.py`) now groups selected accounts **by currency** and runs one replay per currency group (was per-account), so a transfer between two same-currency *accounts* nets (both legs in one replay); conversion is unaffected since a rate depends only on the currency. **Depends on ADR-052:** the funds were renamed on the move, so each transfer's two legs were recorded under two `security` records (e.g. "CRA Qualified Investment" → CRANX; two EMNT records; KLD 400 → DSI) — those must be **merged** first for the pen to match cross-record legs. Options rejected: leaving ShrsOut as a sale (the bug — −$265k of phantom portfolio losses); skipping ShrsIn/ShrsOut like XIn/XOut (positions would be wrong — these carry real share quantities); full persisted cross-currency transfer-linking (round-4, deferred — the pen is the lightweight same-currency form). **Verified** on the live DB: TSBIX total return unchanged at **$19,865.56** but realized **−$52,865 → +$11,593** / unrealized **−$64,492 → +$33** (total-return invariance proves the change is conservative); EMNT held cost **$2,109 → $59,474** and CRANX **$1,630 → $51,532** once their pairs are merged; non-transfer securities are byte-identical (code path unchanged when `is_share_transfer` is False); the Stock Record was already correct (single cross-account replay).

### ADR-054 — Stock splits adjust the FIFO lots
Closes the ADR-044 deferral of stock splits, prompted by **HDV's 5-for-1 split**: Tiingo stores real prices (~$140 pre-split → ~$27 post-split), but the engine never scaled the share count, so it valued 214.779 pre-split shares at the $27.54 post-split price → a bogus −72.7% "loss" (and corrupted the Stock Record, the Returns Unrealized column, and the value chart). The split had been recorded not as a `StkSplit` but as an **`XIn` of 859.116 shares @ $0** (= 214.779 × 4, the extra shares of a 5:1), which the engine ignores. Decision: model a split as a **`StkSplit` action whose `quantity` is the ratio** (new shares per old — 5 for 5:1, 0.1 for a reverse 1:10); a shared `_apply_split` scales every open FIFO lot (`qty ×ratio`, `unit_cost ÷ratio`, **total basis unchanged**) in all three engines (`compute_holdings_view`, `compute_returns`, `compute_value_history`), so market value stays continuous (shares step up as price steps down) with **no price rewrite**. The investment dialog (ADR-048) gains a "Stock split" action with a single "Split ratio" field (hides qty/price/amount, keeps security; seeds the ratio in edit mode). Rejected: rewriting stored prices to be split-adjusted (fights the Tiingo feed); auto-detecting `XIn @ $0` as a split (XIn is a legit transfer). The owner's HDV row is converted by editing it (Action → Stock split, ratio 5). **Verified** on the live DB: HDV 214.779 → **1,073.895 shares**, cost unchanged **$21,655**, avg **$20.17**, market value **$29,575**, unrealized **−$15,740 → +$7,920**; offscreen Qt confirmed the dialog's split mode, save (`StkSplit`/qty=ratio/price=NULL/amount=0), and edit-mode ratio seeding.

### ADR-055 — Net Worth converts to a display currency
Fixes a materially wrong headline: with 7 GBP + 7 USD accounts and (at the time) 0 fx rates, `NetWorthWindow` summed `compute_account_values()` (which returns each account's value in its *own* currency) as bare pence and labelled it "£", so the ~$300k USD brokerage was added to GBP **at par**. ### ADR-082 — Date & period standardisation: one period vocabulary, one date-edit factory
First engineering bite of the 1.0 launch plan's **P1**. A polish audit found date/period handling fragmented: **3+ preset vocabularies** (register `30d/90d/6m/12m/ytd/all`; reports `quarter/6m/ytd/1y/3y/custom`; investment `ytd/1y/3y/5y/max/custom`; Sankey `ytd/mtd/last_month/custom`), **period-bounds maths duplicated in four places** (`account_summary.period_bounds`, the register's `_current_since`+`_months_before`, and inline `_resolve_bounds` in the Sankey + Investment-Returns windows), **the `_PERIOD_LABELS` dict copy-pasted into ~10 files**, and a **latent bug** — the register computed `6m`/`12m` calendar-accurately (`_months_before`, same day-of-month) while `account_summary.period_bounds` used day-deltas (`180`/`365`), so `"6m"` silently meant two different windows depending on the screen. Plus two date display formats across ~14 `QDateEdit` sites (ISO on entry forms, `d MMM yyyy` on a couple of human dialogs, and `schedule_dialog` with no calendar popup). Since the preset keys are **persisted in saved-report `filters_json`**, consolidation had to keep them byte-stable. **Decision:** a new Qt-free **`mfl_desktop/periods.py`** becomes the single source of truth — one `PERIOD_LABELS` registry, the per-context preset SETS (`REGISTER_PRESETS`/`REPORT_PRESETS`/`INVESTMENT_PRESETS`/`SANKEY_PRESETS`, keys unchanged), one `period_bounds(key, today, *, earliest=, custom_start=, custom_end=)` covering every key (`None` start only for unbounded `all`/`max`; raises on `custom` without dates), plus `months_before`/`period_since`/`fmt_date_range`/`period_label`/`labels_for`/`options_for` — and a Qt **`ui/date_widgets.py`** with `make_date_edit(...)` (calendar popup + the one user-facing `d MMM yyyy` format, decision P1a) and `make_period_combo(keys, current=)`. Rejected: one preset set everywhere (the sets legitimately differ — `All`/`5y`/`max`/`mtd` belong to different screens — and it would break persisted keys); hand-syncing the duplication (that drift *was* the bug). **Verified:** `periods.py` unit tests (every key's bounds, the 31 Mar→Feb day-clamp, custom/all/max edges, `period_since`); compute-layer delegation parity + saved-filter round-trip; whole-app import; offscreen construction of the spending + investment filter dialogs (period combos build, `5y`/`max` resolve from the shared registry) and the register preset/since logic. **Follow-up "P1 part B":** roll `make_date_edit` across the ~14 `QDateEdit` sites (incl. the no-popup `schedule_dialog`) to finish the one-date-format half, and adopt `make_period_combo` in the filter dialogs to retire the last label references.

### ADR-080 — Enable Banking production activation & application-key custody (planning; deferred)
Frames decision **F1a** of the 1.0 launch plan without locking it. ADR-077 shipped the Qt-free BYO provider cores (incl. Enable Banking, the free UK/EU Open Banking feed); **automated feeds are a 1.1 fast-follow, so nothing here blocks 1.0.** The path to **production** does need framing: Enable Banking "full activation" requires a contract, **KYC**, a billing account, an end-user-visible app description, a data-protection email, and **privacy-policy + ToS URLs** — none of which exist until workstreams B (company/legal) and W (website) land (and those URLs are launch deliverables anyway, so they double-count). The load-bearing open question: **who holds the EB application registration + private key (`.pem`)?** Options: **BYO** — each user registers their own EB app and supplies the key (what the cores assume; max privacy, MFL is a local tool and **not** a data controller / regulated entity, lightest compliance; cost = one-time setup friction); **MFL-hosted shared registration** — one registration, click-to-connect UX, but MFL becomes data-controller / TPP-adjacent with heavier UK/EU GDPR + KYC + PSD2 obligations and server-side key custody, likely **reopening subscription** (ADR-079's escape hatch); or a **hybrid** (BYO default + optional hosted convenience later). **Decision: defer the custody choice to after the B3 compliance read; ship 1.0 with the BYO cores unchanged.** Constraints that hold either way: the key **never** lives in the binary or the `.mfl` (BYO → OS keychain, the F2 backlog item; hosted → server custody with its own review); EB **consents expire ≈90 days** → a re-consent flow is required (F2); **activation path = Restricted mode first** (whitelist the owner's own accounts so the full consent→fetch→`stage_feed`→review→commit round-trip is owner-tested before any manual review) **→ then Unrestricted** (EB manual review + KYC + contract + billing), gated on B1/B2/B3; the non-sandboxed distribution choice (ADR-078) keeps local custody viable. Trigger to make the call (and write it up as an ADR-080 amendment): **B1 (entity) + B2 (privacy/ToS live) + B3 (PSD2/AISP role determination) done** — i.e. once we actually know whether the hosted model carries regulated-entity obligations we'd take on. Until then, **don't architect against either option** — keep the cores BYO-compatible and credential-source-agnostic. Consequences: 1.0 unblocked; the activation path de-risked by Restricted mode; the F2 work (keychain + re-consent) is correct under either future, so not wasted. Trade-offs: BYO setup friction persists for 1.1; the decision is genuinely blocked on **legal** input, not engineering; a later hosted choice cascades into data-controller status + subscription.

### ADR-079 — Licensing & pricing: one-time perpetual, everything included, offline key
Locks decision **C1** of the launch plan. The decisive fact: MFL is **local-first with BYO feed credentials (ADR-077) — no ongoing server or per-user cost to amortise** (the app only talks to third-party APIs the *user* holds keys for), which removes the usual justification for a subscription. **Pricing: a single one-time perpetual purchase (~£25–45 / $30–50), everything unlocked** — all feed providers (incl. SimpleFIN/Plaid), all reports, investments, multi-currency — with **2.0 as a new paid upgrade** (1.x keys keep working on 1.x forever). This supersedes the earlier "willing-to-pay unlocks SimpleFIN/Plaid" framing: all providers are in the box; the user still supplies their own provider key/cost under BYO. **Enforcement = an offline signed license key, no activation server:** the app ships a public key, a purchased license is a token **signed by our private key** (held by the MoR / fulfilment) and validated entirely on-device — fits local-first + privacy, needs no backend; the key encodes buyer name/email + an **edition/version entitlement** so a 2.0 upgrade is a new key. New **Qt-free `licensing.py`** (verify-only, never holds the private key) + an Enter-License/Buy flow + About-box state; a time-limited **full-feature trial** unlocked by a key. **Payments via a Merchant-of-Record** (Paddle / Lemon Squeezy / FastSpring), not bare Stripe — the MoR is the seller of record and handles **UK/EU VAT + US sales tax + invoicing + key delivery**, the single biggest tax-compliance relief for a solo founder (cross-ref ADR-078: direct distribution means we'd otherwise owe cross-border tax). **Stores (1.1+):** store policy may mandate the store's IAP for digital unlocks (15–30% cut, no out-linking) — the license *model* is unchanged, a store purchase grants the **same entitlement**; reconciled when K3 starts. Rejected: **tiered unlock** (base + paid "Plus" for premium providers — with no marginal cost per provider there's nothing to meter, and it tangles with store IAP); **subscription** (churn-prone with no recurring cost — *only* defensible if MFL ever takes on hosted infra per ADR-080's hosted model, kept as an explicit escape hatch). Trade-offs: offline keys are crackable (accepted — honest non-technical audience; gentle friction, not DRM fortress; online activation can be added if piracy bites); no recurring revenue (fine — no recurring cost); version-entitlement bookkeeping; the MoR takes ~5%+. Ongoing: protect the signing private key; set the 2.0 upgrade price when 2.0 is real; keep the entitlement check store-agnostic.

### ADR-078 — Packaging & distribution: direct signed + notarised downloads first; App Stores deferred
Locks decision **K0** of the launch plan, **closes ADR-050 Tier-3** (packaging — "its own round; own ADR when started"), and **amends ADR-003** (settles the deferred PyInstaller-vs-Briefcase tool choice + "signing TBD"). The owner's "notarised … so it's seen as legit and safe" conflates two separable things: **trust** (a Developer-ID-signed + **notarised** `.app`/DMG passes Gatekeeper; a **code-signed** Windows installer seasons SmartScreen — **neither needs an App Store**) versus **store presence** (badge + auto-update + store-handled payment/tax, at the cost of review, a 15–30% cut, and — on the Mac App Store — **sandboxing**). The sandbox is **fundamentally at odds with MFL's file model** — arbitrary `.mfl` files anywhere (ADR-016), `Snapshots/`+`Library/` written *beside* the live file (ADR-057), the AppData default + cwd bridge (ADR-050 Tier-2) — without security-scoped bookmarks for every user file and relocating the sidecars into the container, a substantial rework. **Decision: ship 1.0 as direct, signed, notarised downloads from our own website; defer both App Stores to 1.1/1.2 as a separate channel** — removing the sandbox work from the 1.0 critical path (the biggest scope lever in the plan). **Toolchain: PyInstaller per-OS** (consistent with ADR-003/MRL; Briefcase considered for its built-in dmg+sign+notarise automation but rejected to keep one toolchain we already run, wiring signing in ourselves); **macOS universal2** (Apple Silicon + Intel) `.app` → `codesign` → `notarytool` → staple → signed DMG, **Sparkle** appcast auto-update; **Windows x64** signed installer (Inno Setup) or signed **MSIX**, **WinSparkle**/App-Installer update feed; Apple Developer Program **$99/yr**, Windows cert **~$200–400/yr** (or Azure Trusted Signing), **hardened runtime** with minimum entitlements (outbound network for FX/prices/feeds + user-selected file access, **no `app-sandbox`**); a reverse-DNS bundle id added to the existing ADR-050 `setApplicationName("MFL")`. Linux/AppImage (from ADR-003) stays possible but is **out of 1.0 scope**. Stores deferred with the rework documented (sandbox entitlements, security-scoped bookmarks, relocate sidecars, store IAP grants the same ADR-079 entitlement). Rejected: **stores day-one** (puts sandbox rework on the launch critical path + concedes the cut before the product proves it sells); **both in parallel** (pays the direct-signing *and* sandbox costs at once for no launch benefit). Consequences: full trust without store gatekeeping, zero sandbox rework, file model retained, one build toolchain. Trade-offs: we run our own signing/notarisation/update infra; Windows cert cost + SmartScreen seasoning lag for a new publisher; no store auto-discovery (distribution leans on the website); we handle payment/tax → the ADR-079 Merchant-of-Record. Ongoing: renew certs, notarise every macOS release, keep update feeds current, hold the ADR-050 portability line, and keep new file-access code sandbox-aware so the future store port isn't made harder.

### ADR-081 — MFL → MRL data exchange: file-based RDF export over the shared ontology
Closes the MRL-integration gap in the 1.0 launch plan (workstream M). MFL (SQLite ledger) and **My Retirement Life** (RDF/Oxigraph retirement + tax projection) are sold as a bundle and must interoperate. ADR-009 already retired the "shared database" idea (ADR-001) for a **data-exchange boundary**, and ADR-006 kept MFL's accounts/person on **`mrl:`-namespaced IRIs** precisely so this would be cheap. Owner decisions (2026-06-16, `AskUserQuestion`): integration is a **1.1 fast-follow**, direction **MFL → MRL**, mechanism **file-based RDF export**, commercial relationship **bundle**. Division of labour: **MFL = financial reality today; MRL = projects it forward** (growth/drawdown/retirement/tax assumptions). The MRL ontology already models the entities MFL owns the data for (`mrl:Account`/`CashAccount`/`InvestmentAccount`/`CreditCardAccount`/`PropertyAsset`, `mrl:Person`, `mrl:IncomeSource`, `mrl:Currency`) and even annotates `mrl:PropertyAsset` *"Activated in v1.0.1 for use by My Finance Life."* **Decision:** a user-initiated, one-way, **point-in-time** Turtle snapshot of **balances + entities, not the transaction ledger** (MRL projects from current balances, not 35k rows). Mapping: `mrl:Person_1` → firstName/lastName/baseCurrency; each account → its stored IRI **verbatim** with `accountName`/`accountBalance` (from `compute_account_balances`/`compute_account_values`)/`balanceDate`/`accountCurrency`→`mrl:Currency_<CODE>`/`isLiability`/`creditLimit`/`accountType`→skos concept/`ownedBy`→`mrl:Person_1`/`exchangeRateToBase`+`Date` (ADR-055 nearest rate, only when account ccy ≠ base). **IRI-class vs rdf:type:** MFL mints `mrl:SavingsAccount_N`/`PropertyAccount_N`/`VehicleAccount_N`, which have **no MRL class** — the IRI is opaque so it's emitted verbatim while `rdf:type` is set to the real MRL subclass (`CashAccount`/`PropertyAsset`/`OtherAsset`). **The authority contract** (the crux of the two-app boundary): MFL is authoritative for, and MRL overwrites on merge, exactly the predicates above; MFL **never emits** the MRL-owned assumptions (`annualGrowthRate`/`annualInterestRate`/`annualDividendRate`/`reinvestDividends`/`drawdownPriority`/`statementDay`/`dateOfBirth`/`targetRetirementAge`/`lifeExpectancy`/`employmentStatus`/`residesIn`/`plansToRetireIn`/`accountJurisdiction`), so a re-import never clobbers the user's MRL inputs. **Mechanism:** Qt-free `mfl_desktop/export/mrl_rdf.py` (`build_mrl_turtle(repo, as_of) -> str`, pure/headless-testable like `fx.py`), a `File ▸ Export for My Retirement Life…` verb, and a CLI `export-mrl`; a **hand-rolled stdlib Turtle writer** (no new runtime dependency, consistent with the post-ADR-009 RDF-free stance) validated in CI by parsing with `rdflib` (test-only) + an Oxigraph round-trip — ⚠ open sub-decision to ship `rdflib` at runtime instead if escaping/prefixes prove fiddly. The `mrl-ontology.ttl` version is a **pinned contract** (ADR-005: never edited from MFL); shared-ontology changes are coordinated. **Phase 1** (the M2 deliverable): person + currencies + accounts. **Phase 2:** `mrl:IncomeSource` derived from `scheduled_txn` income rows (annualised), property metadata, jurisdiction. Rejected: shared DB / live read of MRL's Oxigraph (storage coupling, must locate the store, fragile, breaks under the 1.2 App-Store sandbox), JSON/CSV (discards the shared ontology + aligned IRIs that make this nearly free; MRL is RDF-native). Consequences: cashes in ADR-006 (1:1 account mapping, no matching heuristics), no new shipped dependency, decoupled/one-way/privacy-clean (a user-produced file, easy to disclose in the privacy policy), survives the future sandbox. Trade-offs: one-way + snapshot-only (no projections back into MFL → later M3; no balance history); the merge/authority split must be honoured on the MRL side (recorded here as a joint contract + round-trip-tested); shared-ontology version coupling; IncomeSource/jurisdiction deferred. **ADR-006's IRI-namespace discipline is now load-bearing for interop**, reinforcing pitfall #3. Status **Proposed** — to be Accepted when M2 implementation begins.

### ADR-077 — Bank feeds (Arc H): pluggable providers, GoCardless first
The last arc, and the owner's make-or-break feature: pull transactions automatically instead of exporting + importing files. There is **no free, universal, private bank-feed API**, and the owner's accounts span **UK** (HSBC, Smile, YBS, Capital One UK…) *and* **US** (Ally, Chase, Discover, eTrade, Morgan Stanley…), which no single free mechanism covers — OFX Direct Connect (free/local, US-leaning), GoCardless Open Banking (free, UK/EU only), SimpleFIN (cheap, US), Plaid/TrueLayer (broad but paid). Two constraints follow: a freely-shared app **can't bear central per-user cost**, so each user supplies their own credentials → a **pluggable provider framework**; and a feed only needs to *produce transactions*, so it **reuses the entire existing import pipeline** (staging, FITID/hash dedup, the manual-match heuristic, the review dialog, commit). Owner decisions (`AskUserQuestion`): **GoCardless first** (free, covers their UK banks), and a **manual "Update accounts"** refresh (downloaded transactions land in the existing review/dedup before commit), with background/scheduled refresh deferred. Round 1 (this commit) is the headlessly-testable foundation: `mfl_desktop/feeds/` (Qt-free, stdlib `urllib`, no new dependency) with `gocardless.py` — base `https://bankaccountdata.gocardless.com/api/v2`, `POST /token/new/` (secret_id+secret_key → 24h access), `GET /institutions/?country=GB`, `POST /requisitions/` (hosted bank-consent `link`), poll `GET /requisitions/{id}/` for `status == "LN"` + account ids, then `/accounts/{id}/transactions|balances|details/`; the desktop has no web server so consent opens in the system browser and completion is detected by **polling** (the redirect is a harmless localhost URL — no firewall-prompting local server). `normalize.py` maps a GoCardless booked transaction to the parser raw-txn dict (`bookingDate`→date, abs `transactionAmount`→amount with sign→tx_type, creditor/debtor/remittance→payee, `transactionId`→`fitid`); pending rows are skipped in v1 (they mutate and would churn the dedup). **Migration 0026** adds `feed_account` (MFL account ↔ provider account, one feed per account, `ON DELETE CASCADE`; `UNIQUE(provider, external_account_id)`); the GoCardless `secret_id`/`secret_key` persist in the `setting` table (ADR-035, "stored inside this file" disclaimer) and the short-lived access token is kept in memory only. `Repository` gains a `FeedAccount` dataclass + `link/get/list/unlink/mark_synced/set_status` (link upserts on the account); `ImportService.stage_feed(account_iri, raw_txns)` runs feed rows through `_classify_and_stage` so dedup (`fitid`→`import_hash`) makes a re-fetch of the overlapping window idempotent. A `cli.py feeds-check --secret-id … --secret-key …` probe lists the user's bank institutions so connectivity (the make-or-break) can be proven before any UI is built. Rejected: Plaid/TrueLayer (central per-connection billing can't be sustained for a freely-shared app), catching the OAuth redirect with a bundled local web server (polling is simpler, no firewall prompt), auto-committing fetched transactions (they go through the same human review as file imports), persisting access/refresh tokens (re-minted per session from the secret). **Verified** offscreen (no live network): the normalizer maps booked/pending correctly and drops malformed rows; the GoCardless client mints the token first without auth, sends `Authorization: Bearer` on subsequent calls, builds the requisition body correctly, parses status/accounts/transactions, and turns an HTTP 401 into a `GoCardlessError`; `feed_account` CRUD round-trips, upserts on re-link, and cascades when its account is deleted; and `stage_feed`→`commit_import` imports 2 rows then dedups a re-fetch to 0 imported / 2 skipped (idempotent). Live bank connectivity is verified by the owner against their own GoCardless secret. **Deferred:** the Manage ▸ Bank Feeds UI — connect a bank (browser consent + poll), link provider accounts ↔ MFL accounts, **Update accounts** → review → commit, balances + last-synced (H2); re-consent on ≈90-day expiry, error/empty states, scheduled/background refresh, and a US provider (OFX Direct Connect / SimpleFIN) (H3).

### ADR-076 — Design tokens + light/dark theming (Arc B round 1)
First Arc B round, amending ADR-026's light-only baseline. The per-window styling allowed by ADR-026 had sprawled — ~15 UI files hardcoded **hundreds** of hex literals — and drifted: the baseline used a *gray* ramp (`#6b7280`/`#374151`) while the feature windows used the true Tailwind *slate* ramp (`#64748b`/`#334155`/`#0f172a`), so there were two "slate-500"s plus casing mismatches, and no way to add a dark theme without editing every file. Owner decisions (`AskUserQuestion`): the design-system foundation **and** a dark-mode toggle. The safety discipline that makes a ~140-site refactor low-risk: **every token's light value equals the hex it replaced**, so light mode is unchanged by construction and only the dark values are a fresh design (the gray ramp is unified onto slate as the intended consolidation). Live switching uses two mechanisms re-run on a theme change: (1) `theme.py` rebuilds the QPalette + global QSS from the tokens, re-theming every standard widget (windows, dialogs, inputs, buttons, menus, tables, trees, scrollbars) for free; (2) a **themed-stylesheet registry** — `tokens.themed(widget, "color: {muted}; font-size: 11px;")` formats the template with the active tokens, sets it, and stores `(widget, template)` in a `WeakKeyDictionary`, re-formatting every registered widget on switch (handles arbitrary colour+size+weight combos a fixed CSS-class vocabulary couldn't; `_format` substitutes `{token}` via regex so literal QSS `{ … }` blocks pass through untouched). `tokens.notifier.changed` is emitted so paintEvent charts can repaint. The toggle (`theme.apply_theme(app, name)`) persists `ui_theme` to the `setting` table (no migration); `__main__` reads it on launch (before any window shows, so no flash); a **View ▸ Dark Mode** checkable menu in the register window drives it live. The conversion was done by two bounded regex sweeps (single-line, then multi-line concatenated-literal `setStyleSheet` calls) mapping each hex to its token, plus hand edits for `f-string`/dynamic colours; the register, sidebar, Home dashboard, all dialogs, and the saved-report windows (Spending / Income & Expense / Payee / Category×Payee / Net Worth / Sankey / Investment Returns) theme fully. **Intentionally deferred to B2 as readable light "islands":** the paintEvent charts (they fill a white plot background and draw dark text on it) and the per-account Summary window (`account_summary_window`, a paintEvent dashboard on module-level `_COLOR_*` constants) — both stay internally consistent and legible (dark text on their own white surface) in dark mode; theming them onto the tokens is the next sweep. Rejected: a fixed `QLabel[cssClass="muted"]` vocabulary (the app's inline styles mix colour with size/weight/letter-spacing in too many combinations), apply-on-restart dark mode (worse UX), and keeping the gray ramp (perpetuates the two-ramp split). **Verified** offscreen on a copy of the live database: tokens resolve per theme (`text` `#0f172a`↔`#f1f5f9`, `canvas` `#f8fafc`↔`#0f172a`); a registered probe label restyles on toggle; the register window (with its Dark Mode action checked) and the Spending, Payee, and Net Worth report windows and the Rules dialog all build in dark mode without error; and the toggle round-trips the `ui_theme` setting light↔dark.

### ADR-074 — Capture a scheduled transfer's destination at post time
Owner-reported bug: posting a transfer-kind scheduled transaction that had no destination account failed with *"Transfer schedule is missing a destination account."* (raised by `post_scheduled_txn`) and gave no way to fix it. Setup normally captures the destination — `ScheduleDialog` reveals an inline picker and validates it whenever the category is transfer-kind — but a schedule can become transfer-kind *without* one if its category's kind was switched to `transfer` after the schedule was created (the stored `transfer_to_account_id` stays NULL while `category_kind` flips), or if it was seeded from a transaction without one. The launch auto-post sweep already swallows such failures per-schedule, so the only visible breakage was the manual Post Now flow. Fix (owner's suggestion — capture at setup *or* posting): `SchedulesDialog._on_post_now`, before the amount prompt/confirm, detects a transfer-kind schedule with no destination and opens the existing `TransferDestinationDialog` (unlocked, so any non-source account can be picked; explains and stops if there's no other account), persists the choice via new `Repository.set_scheduled_transfer_destination(schedule_id, account_id)` (validates dest≠source), re-reads the schedule, and continues — so the confirm shows the chosen destination and the existing cross-currency block still collects the partner amount when the currencies differ. Persisting (rather than a one-off destination for this post) repairs the template, so future manual posts *and* the auto-post sweep work and the Schedules list shows the destination. No schema change, no migration; reuses the same destination widget as the register's inline-transfer and cross-currency flows. Rejected: post-once-without-persist (re-prompts every occurrence, auto-post keeps skipping it), fixing only setup/edit (doesn't help the user who hits the error at post time and doesn't know the schedule is mis-configured). **Verified** offscreen: a schedule inserted with `transfer_to_account_id = NULL` raises on post, then after `set_scheduled_transfer_destination` posts both halves with a shared `transfer_id` and correct signs, and dest==source is rejected; offscreen Qt drives `_on_post_now` with the destination dialog stubbed and confirms the destination is persisted, both halves post, and `schedules_changed` fires.

### ADR-073 — Auto-categorisation rules engine (Arc G round 2)
Round 2 of the auto-categorisation arc (ADR-028 round 3), and the screen the owner sketched: *"a dedicated screen to be able to see all of the aliases, and be able to create or delete rules — 'if contains xxxx then payee yyyy', or starts with / ends with / is exactly."* ADR-072 (round 1) covered the *exact-payee* case (a payee remembers its category); this covers the *pattern* case (a stable substring of a varying merchant string). The `rule` table has sat unused since migration 0001; its `pattern_kind` CHECK only allowed `substring`/`regex`, which doesn't match the owner's vocabulary, so **migration 0025** recreates the table with `CHECK(pattern_kind IN ('contains','starts_with','ends_with','is_exactly'))` (the report-CHECK recipe; the table was empty so the CASE-mapping copy is defensive). Aliases and rules stay **separate at the data layer** (ADR-028 Option C / ADR-072 — identity vs. automation); the "unified screen" is a view. The engine is a pure, dependency-free `rules_engine.py`: `rule_matches(rule, payee_text, memo)` (case-insensitive, the four kinds, empty-pattern never matches) and `apply_rules(rules, payee_text, memo)` (priority ascending = highest first, ties by id; **field-independent first-win** so a payee-only rule and a category-only rule compose) — it takes duck-typed rule objects so the Repository's `RuleRow` works without a circular import (mirrors `budget_calc`/`goal_calc`). Applied in `ImportService.commit_import` **after** the ADR-072 alias resolution, loaded once per batch: a rule that sets a payee **overrides** the resolved one (and the per-payee default is re-read from the rule's payee); category precedence when the source left the row Uncategorised (and it isn't a split) is **rule category → else per-payee default**, a file-carried category still winning; `rule_payee` applies to split parents (payee is parent-level) but `rule_cat` doesn't (the parent stays Uncategorised, its lines carry categories). Re-import dedup is unaffected (cash hashes on `payee_raw`). Manual entry runs no pattern rules (no raw text — you pick a payee); instead `NewTransactionDialog` now pre-fills the category from a typed payee's remembered category (optional callback, backward-compatible — the deferred G1 nicety). Retroactive application (owner fork: **ask each time**): on creating/editing a rule the screen offers *"Apply to N matching existing transactions?"*, reusing the same pure `rule_matches` against each transaction's **stored payee name** (the original raw import string isn't retained, so the current name is the proxy — a `contains TESCO` rule still matches a row now named "Tesco") and memo, filling only an **Uncategorised** category / **NULL** payee, never overwriting, never touching transfers or split parents (`count_txns_matching_rule` / `apply_rule_to_existing`). Repository: `RuleRow` (+ display `set_payee_name`/`set_category_path`), `list/create/update/delete_rule` (validates matcher kind/field + at-least-one-setter), `new_rule_iri()`. UI: `rule_edit_dialog.py` (match-field + matcher-kind combos, pattern, optional canonical-payee picker, optional category picker, priority spin; validates pattern non-empty, typed payee resolves, ≥1 setter) and `rules_dialog.py` — **Manage ▸ Rules…**: an editable rules table (When / Sets payee / Sets category / Priority + New/Edit/Delete) and a **read-only aliases** table (the existing `canonical_id` aliases as implicit "is exactly → payee" rules, "manage in Payees…"), giving the owner the whole picture in one place without duplicating alias CRUD. Owner forks (`AskUserQuestion`): rules screen with aliases shown read-only; retroactive ask-each-time; rules set payee and/or category. Rejected: merging aliases into `rule` (conflates identity/automation, disturbs the load-bearing `COALESCE(canonical_id,id)` rollup), regex matchers (power-user feature at odds with the non-technical audience — easy to re-add), matching rules against the resolved payee (that's what the per-payee default already does), rules on manual entry (no raw text), retroactive overwrite (would clobber deliberate edits). **Verified** offscreen: the pure engine (each kind + case-insensitivity, priority + field-independent compose, empty guard); the migration CHECK swap (`is_exactly` present, `substring` gone); Repository CRUD with display names + retroactive scope (only unset/uncategorised, no transfers/splits, never overwrite); an end-to-end Banktivity import (a `contains` rule sets payee + category on a matching row, a file-carried category is kept, rule-category beats the per-payee default, the split guard holds); and offscreen Qt (rule edit round-trip + empty-pattern/no-setter rejection, the rules dialog listing one rule + one read-only alias with the right "Payee text contains …" text, and the New Transaction dialog pre-filling the category from a remembered payee).

### ADR-072 — Payee→category memory (Arc G round 1)
First round of the auto-categorisation arc (ADR-028). After every import the owner re-categorises the same merchants by hand; the app should remember "Tesco → Groceries" and apply it. ### ADR-071 — Investment QIF `P` field is a description, not a payee
Closes the root cause behind the 2026-06-14 live-DB cleanup that deleted 150 junk payees. Investment QIF rows carry a `P` line; in the brokerage exports the owner imports (Banktivity → QIF) it is a free-text **description** ("DIV - SABRA HEALTH CARE REIT INC REC 08/29/25 …"), but ADR-043's round-1 parser copied it into `payee_raw`, and the import service mints a payee from `payee_raw`. So every investment import created one payee per distinct description string, and **re-importing the same QIF regenerated them** — the cleanup was a one-off that the next import undid. The real identity is already captured by the security (`Y` → `security_id`, ADR-044) and the action (`N`); `P` is useful only as memo text. Fix is **parser-only** (`qif_parser._normalise_invst_record`): the memo is now composed from `P` **and** `M` (de-duped so `P==M` doesn't double up) ahead of the existing `Transfer to/from [Account]` note, and `payee_raw` is set to `""` so the service's `get_or_create_payee("")` returns `None` — **investment-action rows never mint a payee**. No schema change, no migration, no Repository/UI change; the cash CSV/OFX/QIF-bank paths don't go through this normaliser and are untouched. **Re-import dedup is unaffected**: investment rows hash on account+date+action+security+quantity+amount (`compute_investment_hash`), never the payee, so previously-imported rows still match and the junk is simply never created again. Owner forks (`AskUserQuestion`): scope = **all investment-action rows** (not just Div/income — `P` is a description for every action type, so gating only dividends leaves the same trap on Buy/Sell/Cash); `P` text = **folded into the memo** (not dropped — no information lost). Amends ADR-043. Rejected: drop `P` entirely (loses the text); gate only Div rows (narrower, same trap elsewhere); a payee-cleanup migration (orthogonal — the one-off already ran; this fixes the *source*); map `P` to the security as payee (conflates two first-class concepts). **Verified** offscreen on a synthetic investment QIF: Div (`P` only) → `payee_raw=''`, memo = the `P` text; Buy (`P`+`M`) → `payee_raw=''`, memo folds both deduped; Cash/transfer (`P`+`L`) → `payee_raw=''`, memo = `P` + the transfer note; confirmed `get_or_create_payee('')` is `None` and the dedup hash excludes the payee.

### ADR-070 — Category lifecycle: archive / restore
Arc D, the sibling of ADR-069 on the hierarchical `category` tree. The only category removal today was `delete_category`, which re-points the category's transactions to **Uncategorised** and drops the row — destructive of the historical attribution. The owner wants to *hide* a category without rewriting history and bring it back later. `category.archived_at` was reserved by ADR-011 and the tree/picker/budget queries already filter `WHERE archived_at IS NULL`; nothing set it. So, **no migration** — mirror the account decisions: one **Archive** concept, excluded-by-default with a "Show archived" toggle, flow-report history retained (reports aggregate `txn`/`txn_category_line` by `category_id`, never gated on `archived_at`). The tree adds one wrinkle: archiving must keep it consistent — no active child stranded under an archived parent. So **archive cascades down** the subtree (`archive_category(id)` archives id + all descendants; rejects Uncategorised id=1; idempotent) and **restore cascades down *and* up** (`unarchive_category(id)` clears id + subtree + every ancestor to the root, so a restored node is always reachable), maintaining the invariant **"every non-archived category has a non-archived parent (or is a root)."** The parent pickers (New / Reparent) refuse archived nodes, and `find_or_create_category_path` **resurrects** (un-archives) any matched archived path on import — the `UNIQUE(parent_id, name)` constraint blocks a fresh sibling, and an import re-using a path means it's in use again. `CategoryNode` gains `archived`; `list_category_tree(include_archived=False)`; new private `_category_ancestors` (the recursive mirror of `category_descendants`). The Categories dialog gains a **"Show archived"** checkbox (archived rows greyed + a "(archived)" suffix) and one **Archive/Restore** button that flips by selection (Restore when every selected row is already archived, else Archive — which confirms, noting the subtree cascade and that history is kept). `delete_category` stays as the destructive variant; the dialog now leads with Archive. Trade-offs: cascade restore can un-archive children individually archived earlier inside a since-archived branch (acceptable for "restore a branch"); reports' category checklists drop archived categories from their *filters* (history still in the unfiltered totals); `category_descendants` / `category_has_children` / `_sibling_exists` deliberately still see archived rows so a name collision with an archived sibling still blocks a rename/create. **Verified** offscreen on a live-DB snapshot: archiving a real 13-node branch cascades to exactly its 13 descendants; the branch vanishes from the default tree and returns with `include_archived=True` (all flagged `archived`); restoring a deep descendant restores its ancestors so it's reachable; full restore brings the branch back; archiving Uncategorised raises; an archived leaf re-imported via `find_or_create_category_path` returns the same id (no duplicate) and reappears; and the dialog's Show-archived reload runs clean.

### ADR-069 — Account lifecycle: close / reopen (soft-close)
Arc C of the lifecycle arc. Accounts accumulate over a financial life and get closed — but the only removal today is the destructive `delete_account`, which cascades away every transaction and all import history. Wrong for a *closed* account: its history is real and should still count in the flow reports; the owner just wants it out of the sidebar and Net Worth. `account.archived_at` was reserved by ADR-011 and every list query already filters `WHERE archived_at IS NULL` — the column was never *set*. So this is almost entirely UX + a verb, **no migration**. Owner decisions (`AskUserQuestion`): **one Close verb** (not Close vs. Hide), and **excluded-by-default with a "Show closed" toggle**. Repository: `AccountSummary` gains `archived_at` + an `is_closed` property; `list_accounts` / `list_investment_accounts` / `compute_account_balances` / `compute_account_values` gain `include_closed=False` (default **unchanged**, so every existing caller — transaction/transfer pickers, budgets, all reports — is byte-identical; only the sidebar build and Net Worth's toggle pass `True`); new `close_account(id)` sets `archived_at = datetime('now')` (idempotent), `reopen_account(id)` clears it, `delete_account` stays as the destructive variant. Sidebar: the register window feeds it `include_closed=True` and it **partitions** open vs. closed itself — open accounts render exactly as before, closed accounts are pulled into one **collapsible "Closed accounts (N)" group** at the bottom of the ACCOUNTS section (collapsed by default, muted slate-400 leaves that stay selectable so a closed account's register is viewable, a `CLOSED_ROLE` marker so the context menu offers the right verbs; `select_account_by_iri` descends into and expands the group). Register window: Account menu gains **Close Account…** (above Delete, enabled only for an open selection); a closed account's right-click offers Summary / **Reopen Account** / Delete, an open account's offers Edit / Move / **Close Account…** / Delete. Net Worth: a **"Show closed accounts"** checkbox threads `include_closed` into every total, donut, and column (closed leaves get a " (closed)" suffix). **Flow reports keep closed history by construction** — Spending / Income & Expense / Sankey / Payee / Category aggregate the `txn` / `txn_category_line` tables and never gate on `account.archived_at`, so a closed account's transactions flow into the unfiltered totals; only the report *account-pickers* (which use `list_accounts`) stop offering it. Trade-offs: you can't *filter a report to* a single closed account (additive per-report toggle if it bites); closing doesn't auto-zero a non-empty account (it just disappears from Net Worth until "Show closed" is ticked). **Verified** offscreen on a WAL-consistent live-DB snapshot (28 checks): close/reopen membership in `list_accounts` + `compute_account_values` (with and without `include_closed`), the Net Worth sum dropping then restoring the closed balance (headline £2,589,980 ↔ £2,656,421), idempotent re-close keeping its timestamp, the sidebar rendering a collapsed Closed group with the `CLOSED_ROLE` marker and `select_account_by_iri` finding the closed account, the Net Worth toggle changing the headline, and the Categories dialog smoke.

### ADR-067 — Net Worth two-ring donut (a scoped exception to "no pies")
Arc E round 3 (E4). The Net Worth window's summary panel used a horizontal **proportional bar** for composition precisely because ADR-018 ruled out pies ("rarely convey accurate data"). The owner asked to replace it with a **two-ring donut**: inner ring = account *type*, outer ring = the *individual accounts* within each type. That conflicts with ADR-018, so it's an explicit decision to **narrow** that rule, not ignore it — the coherent line: pies/donuts stay banned for **time-series / cross-series comparison** (the spending, income & expense, payee, Sankey, returns reports — where a pie is actively worse than bars+line), but a donut is **allowed for a point-in-time composition of one positive whole**, which is the one case it reads well (and the two-ring form adds a type→account level a single bar can't show without a dense stack). The family legend, Assets/Debts columns, and banner are untouched; the summary column was widened a little for the donuts. Reuses the existing standalone Net Worth window — no new saved report type, no migration (the reserved `net_worth` saved type stays unused; a second entry point would only confuse). Owner forks (`AskUserQuestion`): two donuts (over assets-only or one signed magnitude donut), replace the bar in the existing window (over a new saved report), inner ring = account type (over family). `proportional_bar.py` is now unused but left in place (still a valid generic widget). **Amends ADR-018** (status note added there). **Verified** offscreen on seeded temp DBs: the assets inner ring is exactly the three seeded asset types; the Savings type nests its two accounts (ISA + Savings) whose values sum to the type's £8,000; outer children carry distinct shaded colours; the debts donut shows one Credit card type (£500) and is shown/hidden by debt presence; a USD account with no GBP rate is excluded from the donut and appears in the missing-rate list; and the empty + single-segment paint paths render without error (via `grab()`).

### ADR-061 — Register search performance (debounce + cached haystack + coalesced status)
Closes a freeze the owner hit: opening a busy account with **Show: All** and typing in the Search box froze the screen on every keystroke. Profiling the real DB (heaviest account = 13,550 rows on the All window) found three costs running synchronously per keystroke with no debounce: `filterAcceptsRow` rebuilt a 7-field haystack + two `f"{x:.2f}"` amount formats for *every* row (~15 ms/pass); `_update_status` walked every visible row (`mapToSource` + `Decimal` add, ~25 ms) and fired **2–3× per keystroke** because it was wired to `rowsRemoved` + `rowsInserted` + `layoutChanged`; and nothing debounced the typing, so a fast typist stacked ~90 ms passes back-to-back. This is the search-time follow-on to ADR-041's flagged "All is still slow on huge accounts" (ADR-041 made *load/sort* of All opt-in but left *search over* an All view untouched). Fix is all view/model-layer, no migration/schema/Repository change: **(a)** the Search box debounces through a 200 ms single-shot `QTimer` (stash text → restart timer → one `set_search` when typing settles); **(b)** `TransactionTableModel` caches each row's lowercased search blob (`_search_blobs`, index-parallel with `_rows`, built in `reload()` and refreshed in `setData`; `search_blob_at` exposes it), so `filterAcceptsRow` is a single substring test (~15 ms → ~2 ms/pass, verified byte-identical to the old inline logic across all 13,550 rows); **(c)** the four proxy signals route through a zero-delay single-shot timer that coalesces a filter invalidation's burst into one `_update_status` walk. Rejected: server-side SQL search (premature — 2 ms in memory now); disabling `dynamicSortFilter` (forward-typing pays no sort cost, so it targets the wrong case). **Verified** offscreen on the live DB: the heaviest account loads 13,550 rows + 13,550 blobs with 0 parity mismatches; a keystroke leaves the view unchanged until the debounce fires, then filters correctly (13,550 → 99 for "amazon"); the status walk over ~9.9k visible rows is one 25 ms pass.

### ADR-060 — Configurable grandfather-father-son snapshot retention
Amends ADR-057. Prompted by the owner asking to "let people configure snapshots… after 6 months there could be several GBs" once the Data Library screen (ADR-059) made the `Snapshots/` folder visible. The disk worry is inverted — the flat `SNAPSHOT_KEEP = 10` already bounds the folder — but it exposes the real defect: a busy day fires the 30-min timer well past 10 times, so keep-newest-10 holds only the last few hours and **every restore point older than this morning is already gone** (defeating ADR-016's "my data as of last Saturday"). Fix is the owner's own framing — *30-min on the day, then daily for a week, then monthly* — a **grandfather-father-son (GFS)** scheme: keep *all* snapshots within `subdaily_hours`, then *one per calendar day* back to `daily_days`, then *one per calendar month* back to `monthly_months`, dropping older (defaults 24h / 7d / 12mo). Two owner forks (`AskUserQuestion`): **editable tier values** (spin boxes, not presets) and a **capped** oldest tier (not kept-forever), so disk stays bounded (~today's captures + 7 daily + 12 monthly). Four knobs persist per file in the `setting` table (ADR-035) via a `RetentionPolicy` dataclass + `load_policy`/`save_policy`, floor-clamped so junk/zero can't disable retention. `prune` rewritten as `prune(db_path, policy, now)` — walking newest→oldest, keep-all / newest-per-day / newest-per-month / drop; a new `_stamp_of` parses the embedded filename timestamp (tolerating the `-N` collision suffix) and returns `None` for unrelated files so they're never deleted; `maybe_snapshot` prunes on both the wrote-a-snapshot and nothing-changed paths so an aging set self-thins on idle days. UI: a small `SnapshotSettingsDialog` (four `QSpinBox`es + a live plain-English summary) reached from a `Settings…` button on the Data Library Snapshots tab; Save persists + prunes immediately + emits, and the window re-arms its capture timer to the new cadence (and per-file on adopt, since policy travels with a loaded file). No migration (`setting` exists), no new dependency, pure-stdlib date math. **Verified** offscreen: a 14-month, every-6-hours set of 1680 fakes pruned to 25 (5 sub-daily / 7 daily / 13 monthly), nothing kept past the 12-month cap, one-per-day/month uniqueness, idempotent re-prune; policy save/load round-trip + floor-clamp; and dialog build / summary text / `policy_saved` → list-refresh + `settings_changed` relay.

### ADR-057 — Automatic rotating database snapshots + clean-close checkpoint
Closes the "save on close + auto-save" backlog item — but reframed by ADR-050 Tier-2, which had already made the default file a live, auto-committing `.mfl`, so "write the open file on close" was *already true*. Two real gaps remained: (1) we open in **WAL mode** but never checkpointed or closed on exit, so committed-but-uncheckpointed writes sat in the `.mfl-wal` sidecar — copying the bare `.mfl` silently dropped them; (2) auto-commit gives no protection against *logical* mistakes (a bad import persists just as faithfully). Fix: a `RegisterWindow.closeEvent` runs `Repository.checkpoint()` (`PRAGMA wal_checkpoint(TRUNCATE)`) + `close()` so the single file is self-contained; and **rotating timestamped snapshots** (`mfl_desktop/snapshots.py`) copied via `save_copy` into a `Snapshots/` folder beside the live DB on **launch + a 30-min in-session timer + clean close** (owner picked the most-coverage option), keeping the newest `SNAPSHOT_KEEP=10`. `maybe_snapshot` is best-effort (never blocks the UI) and **change-detects via `max(mtime(.mfl), mtime(.mfl-wal))` vs the newest snapshot** — the WAL mtime is the real signal since the main file's only moves on checkpoint — so the three triggers don't churn byte-identical copies. Restore reuses **File ▸ Open** (snapshots are ordinary, schema-upgradeable DBs); `_swap_repository` gives the file being *left* the same checkpoint+snapshot. Config is module constants (Preferences promotion deferred). **Verified** against a temp DB: snapshot written on change / skipped on no-change, rotation to `keep`, same-second collision suffixing, `wal_checkpoint(TRUNCATE)` zeroing the sidecar, and a snapshot opened back as a faithful populated copy; plus offscreen-Qt window construction confirming the launch snapshot, a live 30-min timer, and close.

### ADR-091 — Surface auto-post failures, and guard category-kind changes that orphan schedules
The owner reported an auto-post expense schedule (Polestar 2 → "Asset Depreciation", −£450/mo) that sat overdue for *dozens* of launches and never posted. Refuse, not auto-repair: a destination is a user choice the system can't infer (same conclusion as ADR-074's post-time prompt). Return-type change to `auto_post_due` (only caller updated); no schema change, no migration. **Verified** offscreen on copies of the live file: the sweep returns `posted=[]` + a named failure for the broken schedule (previously a silent empty result); `change_category_kind(90, 'transfer')` is refused and names the schedule; an unrelated kind-change with no dependent schedules still succeeds.

### ADR-092 — Reopen the last-used file on launch
The owner's annoyance: quit + relaunch always reopened the **default** file in the default location, never the file they'd `File ▸ Open`'d and been working in. The launch resolver (`__main__.main`) kept no memory of the runtime open, and the obvious place to store it — the per-file `setting` table — is the wrong layer: it lives *inside* each `.mfl`, but the path has to be known *before* any file is open. So this is app-level, cross-file state, persisted in **`QSettings`** (the OS-standard per-user store: plist / registry / ini, one mechanism, no platform branch per ADR-050 rule 9). New `mfl_desktop/app_session.py` exposes `remember_last_db` / `last_db_path` (a no-arg `QSettings()` keys off the app's org + application names, so `main` now also sets `setOrganizationName(APP_NAME)`); the path is recorded whenever a file becomes the live file — at launch and on `File ▸ Open` (`_swap_repository`) — while a loaded dataset (`_load_dataset`) keeps the same working file and so is already remembered. Launch precedence is now `--db` → **last-opened (if it still exists)** → legacy cwd dev DB (`mfl_dev.mfl`/`.db`) → appdata default; last-opened sits above the cwd dev convenience so "reopen what I was working on" wins, yet a fresh checkout with nothing recorded still gets the dev file (and that first launch records it). A remembered file that exists but won't open (corrupt / not an MFL DB) falls back to the default instead of stranding the launch. No schema change, no migration, nothing written into any `.mfl`. **Verified** offscreen (isolated `QSettings`: round-trip; relative path stored absolute) + offscreen Qt (`_swap_repository` updates the remembered file from A to B; importing `__main__` raises no circular import). **Amended 2026-06-21 (transient-absence robustness):** an iCloud-Drive `.mfl` quit-then-relaunch opened the default instead — `last_db_path()` filtered on `exists()` so a not-yet-materialised iCloud file read as "unset", and the launch then re-recorded the fallback default, *clobbering* the pointer permanently (compounded by `Repository()` recreating any missing path). Now `last_db_path()` returns the recorded path without an existence check; the resolver checks existence itself and, on a remembered pointer it can't use this launch (absent or won't-open), opens the default **for the session only without overwriting the pointer** so the next launch retries once the file is back, and never hands a missing path to `Repository`. Verified offscreen: evict→fall back with pointer preserved; restore→reopen; missing file not recreated.
