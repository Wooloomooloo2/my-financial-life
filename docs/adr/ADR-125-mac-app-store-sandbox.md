# ADR-125 — macOS distribution: Mac App Store (sandboxed), reversing ADR-123's macOS-out-of-scope

**Date:** 2026-06-30
**Status:** Accepted (decision + engineering arc; implementation to follow in increments)
**Amends:** ADR-123 (Microsoft Store-only — *Windows* stays MS Store; this brings **macOS back into scope as Mac App Store-only**). Implements the macOS leg of ADR-078 **K3** (App Stores), which ADR-078 had deferred to 1.1/1.2.
**Related:** ADR-104 (PyInstaller packaging scaffold — `mfl.spec`/`build_macos.sh`). ADR-101 (app `.icns`). ADR-098 (first-run dialog — gains the folder picker). ADR-109 (cloud handling + snapshots relocated to app-data — the precedent for sidecar relocation, and the `brctl` subprocess that the sandbox blocks). ADR-079 (offline licence key + trial — goes dormant in the store build). ADR-050 / ADR-016 / ADR-057 (the file model the sandbox constrains: arbitrary `.mfl` anywhere + sidecar `Snapshots/`/`Library/` beside the file).

## Context

ADR-123 (2026-06-29) pivoted distribution to **Microsoft Store-only for 1.0.1** and explicitly put **macOS out of immediate scope**. The owner has now decided to pursue the **Mac App Store (MAS)** as the macOS channel. This re-opens the one path ADR-078 (K0) and ADR-123 both flagged as the hard one: MAS requires the **App Sandbox**, which conflicts with MFL's file model in three concrete ways the codebase still embodies (audited 2026-06-30):

1. **Reopen-last-file** persists a **plain resolved path string** in `QSettings` (`app_session.last_db_path`, consumed by `launch.resolve_database`). A plain path is *not accessible* on the next launch inside a sandbox — the app only retains access to user-chosen files via **security-scoped bookmarks**.
2. **The Data Library** (`data_library.py`) writes a `Library/` folder **beside the live `.mfl`**. A security-scoped bookmark to a *file* does not grant write access to its **sibling folder**, so the Data Library breaks under the sandbox. (Snapshots already moved off beside-the-file to app-data in ADR-109, so that half is done.)
3. **iCloud on-demand download** (`cloud.py`) shells out to **`brctl`** via `subprocess`; the sandbox blocks spawning helper executables. (Already written to no-op on failure.)

Plus the packaging itself differs: `build_macos.sh` produces a **Developer-ID-signed, notarised DMG** (direct download). MAS needs a *different* pipeline — App Sandbox + user-selected-file + bookmark + network-client entitlements, **Apple Distribution** signing, an embedded **provisioning profile**, a **`.pkg`** built with `productbuild`, and upload via Transporter to App Store Connect.

## Decision

Ship macOS as **Mac App Store-only**, fully sandboxed, **paid up-front**. Three sub-decisions (owner, 2026-06-30):

- **MAS-only on Mac** (no parallel direct Developer-ID DMG channel). Rationale: one shipping artifact = one behaviour mode to support; the Store handles signing, payment, tax, and auto-update. *Note:* the same source still runs **unsandboxed in development**, so the code branches on a runtime **`is_sandboxed()`** check — not on a second shipping channel. Rejected: MAS + direct DMG (two channels, two trust/update stories, ongoing dual support); the Developer-ID `build_macos.sh` from ADR-104 is retained in-tree only as a dev/debug convenience, not a release channel.
- **First-run "choose a folder"** for the working `.mfl`. On first run (sandboxed), the user picks a real, visible folder (e.g. `~/Documents` or iCloud Drive); we create the default file there and persist a **security-scoped bookmark** so it reopens across launches. Rejected: defaulting into the **container's** hidden `Documents` (`~/Library/Containers/<id>/Data/Documents`) — sandbox-legal and zero-friction, but the file is invisible in Finder/iCloud, which is hostile to a finance app the user must be able to find and back up.
- **Paid up-front** (no in-app purchase). The Store is the entitlement; the ADR-079 offline-key + 30-day trial go **dormant** in the store build via a build flag (same reasoning ADR-123 applied to MSIX). Rejected: free-with-StoreKit-IAP-unlock (needs native StoreKit integration + an in-app feature gate for no benefit at this price point).

### Engineering arc (sandbox compliance is the spine)

A — **Sandbox foundation.** New `mfl_desktop/sandbox.py`: `is_sandboxed()` (detect via the `APP_SANDBOX_CONTAINER_ID` env var Apple sets) + a security-scoped **bookmark store** (create/resolve/`startAccessingSecurityScopedResource`) via **PyObjC** (`pyobjc-framework-Cocoa`, a new **macOS-only** runtime dep — Qt exposes no bookmark API). Both degrade to plain-path behaviour when not sandboxed, so dev runs and any non-macOS build are unaffected.

B — **Working-file persistence via bookmarks.** `app_session` stores a bookmark (base64 in `QSettings`) alongside/instead of the plain path; `launch.resolve_database` resolves the bookmark and brackets DB access in `startAccessing…`/`stopAccessing…`. File ▸ Open and the first-run folder picker both mint a bookmark from the powerbox-granted URL (Qt's `QFileDialog` already routes through the macOS open panel = the powerbox, so *picking* already grants session access; only *persistence* needs the bookmark).

C — **Relocate the Data Library to the container** (mirror ADR-109's snapshots move): `data_library` writes under `AppDataLocation` (→ container Application Support when sandboxed) instead of beside the live file, so sidecars never depend on folder-sibling access. Legacy beside-the-file `Library/` is union-read where folder access happens to exist (unsandboxed), best-effort.

D — **Sandbox-aware degradation.** `cloud.request_download` becomes a no-op under sandbox with a "reveal in Finder to download" hint (no `brctl`); the `launch.py` cwd dev-bridge (`mfl_dev.mfl`) is skipped when sandboxed; the legacy beside-file snapshot union-read is guarded.

E — **MAS packaging.** New `packaging/MyFinancialLife.entitlements` (`com.apple.security.app-sandbox`, `…files.user-selected.read-write`, `…files.bookmarks.app-scope`, `…network.client`) wired via `mfl.spec`'s existing `entitlements_file` hook; new `packaging/build_mas.sh` (Apple Distribution signing of every embedded Qt/Python dylib + the app, embed provisioning profile, `productbuild` → signed `.pkg`, Transporter upload). The fiddliest part is signing PyInstaller's embedded frameworks consistently; tracked as the build-out risk.

F — **Store build flag.** A build-time flag puts the ADR-079 licence nag/trial dormant (Store owns entitlement); paid-upfront means no IAP code.

## Consequences

- macOS is back **in scope** and is **MAS-only**; Windows remains **MS Store** per ADR-123. The project now targets two stores, two MoRs, zero self-managed signing/tax — consistent with ADR-123's economics.
- A new **macOS-only runtime dependency** (PyObjC Cocoa) enters the desktop build for bookmarks. It is import-guarded so non-macOS and unsandboxed runs never touch it.
- The file model is **preserved in spirit** (the user still keeps a real, visible `.mfl` they choose) but the *mechanism* changes under sandbox (bookmarks instead of bare paths; sidecars in the container). ADR-016/050/057 are amended in practice for the sandboxed build; the unsandboxed dev build is unchanged.
- **What needs the Apple Developer account vs. not:** all of A–D and the entitlements plist can be **written and unit-tested on a Mac today** without the account; only the *signing/upload* steps of E need Apple Developer Program enrolment (org, gated on the B1 company — see backlog). So engineering can proceed immediately.
- The 1.0 launch backlog's macOS rows (K1 direct-DMG, K3-deferred) are **reconciled**: K1 direct is demoted to dev-only, K3-macOS is promoted to the active macOS plan.
- This is a multi-increment arc; this ADR records the decision and the plan. Each increment (A…F) ships and is verified on its own, and this ADR is amended if an increment forces a design change (e.g. if PyObjC bookmark behaviour or PyInstaller MAS signing demands it).

## Implementation status

- **Increment A — DONE (2026-06-30).** New `mfl_desktop/sandbox.py`: `is_sandboxed()` (via `APP_SANDBOX_CONTAINER_ID` + `darwin`), `bookmarks_supported()`, the `create_/resolve_security_scoped_bookmark` PyObjC primitives, a `ResolvedLocation` scope handle (context-manager `start`/`stop`), and the `encode_location`/`decode_location` policy (real bookmark only when sandboxed; tagged plain path otherwise; bare legacy paths still decode so the existing `session/last_db_path` keeps resolving). Everything degrades to plain paths off-Mac / without PyObjC / unsandboxed and never raises. `pyobjc-framework-Cocoa==12.2.1` added to `requirements-desktop.txt` under a `sys_platform == "darwin"` marker. `tests/test_sandbox.py` 10/10 on both a PyObjC-capable interpreter (real bookmark round-trip exercised) and the base interpreter (native test self-skips). Not yet wired into `app_session`/`launch` — that's increment B.
- **Increment B — DONE (2026-06-30).** Bookmark-based working-file persistence wired into `app_session` + `launch`, *additively* so non-sandbox behaviour is byte-identical. `app_session.remember_last_db` now also mints + stores a security-scoped bookmark (`session/last_db_bookmark`) **when sandboxed** (cleared otherwise); the plain `session/last_db_path` stays the primary pointer, so `last_db_path()` and the resolver's logic are unchanged. New `app_session.begin_main_file_access()` resolves the bookmark and **starts** scoped access, returning a live `ResolvedLocation`. `launch.resolve_database` calls it on the pointer branch (the bookmark's path is authoritative — it tracks a moved file), carries the live location in a new `Resolution.location` field (which `main()` already keeps referenced across `app.exec()`, so access lasts the session), and releases it (`stop()`) on every recovery-swap/quit branch. File ▸ Open mid-session needs no change: the powerbox grants session access to a just-picked file, and `remember_last_db` re-bookmarks it for next launch. Tests: `tests/test_launch_resolution.py` 11/11 (2 new sandbox-bookmark cases), new `tests/test_app_session_sandbox.py` 6/6 (hermetic temp-QSettings; pins the bookmark persistence + `start()` contract), `tests/test_sandbox.py` 10/10; `compileall` clean; the real app boots through the modified launch path offscreen and the integrated dev path returns `location=None` (no regression).
- **First-run "choose a folder"** (the file-location decision) is a small **follow-on UI step** on `FirstRunDialog`/`first_run_default_path`, not part of B's persistence plumbing — tracked separately.
- **Increment C — DONE (2026-06-30).** The Data Library moved off beside-the-file to the app-data root, mirroring exactly what ADR-109 did for snapshots. New `app_session.library_root()` (key `locations/library_root`, default `AppDataLocation` → the container under sandbox). `data_library.library_dir()` now resolves under `library_root()` via a lazily-imported `_library_root()` indirection (keeps the module Qt-free), the written folder is renamed `Library` → **`MFL Library`**, and `list_saved()` **unions** the legacy beside-the-file `Library/` folder (read-only, de-duped preferring the new location) so no upgrader's saved datasets are orphaned. `clone_database`/`delete_file`/`rename_saved`/`current_file` operate on explicit paths and are unchanged. No new relocation UI (a `set_library_root` + Locations-dialog entry is an optional follow-on; the default covers every build). Tests: `tests/test_data_library_current.py` 6/6 (relocation + legacy union + dedupe), green on both interpreters; `compileall` clean; offscreen the `DataLibraryDialog` builds and lists against the relocated folder (`~/Library/Application Support/MFL/MFL/MFL Library`).
- **Increment D — DONE (2026-06-30).** Sandbox-aware degradation of the two spots that misbehave inside the sandbox. `cloud.request_download` now **skips the `brctl` subprocess when sandboxed** (`sys.platform == "darwin" and not sandbox.is_sandboxed()`) and falls through to the generic read-to-hydrate, which works for the file we hold access to (bookmarked working file / powerbox pick) — a sandboxed app can't spawn a helper. `launch.resolve_database` **skips the cwd dev-bridge** (`LEGACY_DB_CANDIDATES` / `mfl_dev.mfl`) when sandboxed, since the cwd is unreadable and a relative path can't be opened there — a sandboxed first launch with no pointer goes straight to the first-run default. The legacy beside-file union-reads (snapshots + library) need no guard: they're gated on `.is_dir()`, which simply returns False without folder access. Tests: `tests/test_cloud.py` 8/8 (+1 sandboxed-skips-brctl, darwin-gated), `tests/test_launch_resolution.py` 12/12 (+1 sandboxed-skips-legacy-cwd); no import cycle (cloud→sandbox, launch→cloud+sandbox), Qt-free modules still import on the base interpreter.
- **Increment E — DONE (scaffold, 2026-06-30).** MAS packaging artifacts, env-gated like `build_macos.sh` so they no-op cleanly without the certs. New `packaging/MyFinancialLife.entitlements` — the minimal sandbox set (`app-sandbox`, `files.user-selected.read-write`, `files.bookmarks.app-scope`, `network.client`); sidecars are in the container (C) so no beside-file entitlement is needed. New `packaging/build_mas.sh` — reuses `mfl.spec`, embeds the provisioning profile, signs **inside-out** (nested dylibs/frameworks first, then the bundle with `--entitlements`; `codesign --deep` is disallowed for MAS), builds a signed `.pkg` via `productbuild`, and uploads to App Store Connect via `xcrun altool` when API-key vars are set (else points at Transporter). `mfl.spec` gets a one-line comment noting macOS signing is script-side. **Decision deviation from the plan:** entitlements are applied at *codesign time in the script* (not via the spec's `entitlements_file` hook), matching how `build_macos.sh` already signs post-build — more reliable than PyInstaller-native MAS signing. **Verified without the Apple account:** `plutil -lint` on the entitlements (OK), `bash -n` on the script (OK), the spec still compiles, and the script is `chmod +x`. **Needs the owner's Mac + Apple Developer enrolment to actually run** the signing/`.pkg`/upload (and to confirm PyInstaller's embedded Qt/Python frameworks sign cleanly for MAS — the known-fiddly part). CI is unchanged (MAS needs signing secrets it can't have in public CI). **QA item:** confirm report exports go through a save panel (powerbox) so no extra `files.downloads` entitlement is required.
- **Increment F — DONE (2026-06-30).** Store-build licensing dormancy. New `version.is_store_build()` resolves via a stamped `_build_info.STORE_BUILD` flag (set by `stamp_build_info.py --store`, which `build_mas.sh` now passes) and falls back to `sandbox.is_sandboxed()` (the MAS build is the only sandboxed one), so an unstamped MAS run is still treated as a store build; a dev checkout is `False`. `license_service.current_status()` — the single chokepoint every UI surface reads — short-circuits in a store build to an unlocked `STATE_LICENSED` ("Purchased via the App Store") **without starting the trial clock**, so the launch nag and the trial title-cue go silent for free. The Help-menu **Enter-License + Buy** actions are hidden and the **About** box renders a plain "Purchased via the App Store" line with both action buttons hidden (Apple forbids external purchase links; there's no key to enter). Tests: new `tests/test_store_build.py` 6/6 (flag resolution + chokepoint, dev path unchanged); full sandbox-arc regression green (sandbox 10 / app_session 6 / launch 12 / data_library 6 / cloud 8); `compileall` clean; an offscreen smoke in a real `STORE_BUILD=True` stamp confirmed: owned status, empty title suffix, zero License/Buy actions in the window, silent nag, About store-line with hidden buttons (dev stamp restored after).

## Engineering arc complete (A–F)

All six sandbox/packaging increments are implemented + tested as of 2026-06-30. The app is now correct inside the macOS App Sandbox while byte-identical in dev / non-Mac, and the MAS packaging is scaffolded. **What remains is non-code / account-gated:**
- **Apple Developer Program enrolment** (org, gated on the B1 company) → the certs + App Store provisioning profile → then `build_mas.sh` can actually sign + build the `.pkg` + upload. This is also where PyInstaller's embedded Qt/Python frameworks must be confirmed to sign cleanly for MAS (the known-fiddly part).
- **First-run "choose a folder" UI step — DONE (2026-06-30).** `Resolution` gains a `first_run_default` flag set only on the unattended first-run-default branch (not `--db`, not a recovery pick). `__main__` adds `_prompt_first_run_location()` — shown over the splash, **only when `res.first_run_default and sandbox.is_sandboxed()`** — a brief info box then a folder picker (the macOS powerbox, which grants session read/write to the chosen folder); the new `.mfl` is created there and `remember_last_db` mints its security-scoped bookmark. Cancel falls back to the container default so a hesitant user still gets a working app. The dev / direct build is untouched (the `is_sandboxed()` guard skips the prompt → ADR-109 `~/Documents` default). Tests: `test_launch_resolution` 12/12 (now asserts `first_run_default` True for the default, False for a recovery pick); offscreen smoke of `_prompt_first_run_location` (pick → chosen-folder path; cancel → default).

## Addendum (2026-06-30) — first real ad-hoc sandbox build: hardening + the iCloud/SQLite finding

Building and running an actual **ad-hoc-signed sandboxed** app on the owner's Mac surfaced two things the plan didn't anticipate. Code changes below are in the working tree **but not yet committed** (the A–F arc was committed in `4761dfc`; these are follow-ups).

**1. Local-build hardening (`packaging/build_mas.sh`).**
- Added a **`--adhoc`** mode: signs with `codesign --sign -` + the sandbox entitlements (no Apple account), skips the `.pkg`/upload, and drops the result in `~/Applications`. This is how the sandbox is tested locally without enrolment — the OS enforces the sandbox for any signed app carrying `com.apple.security.app-sandbox`.
- **iCloud xattr blocker:** the repo lives under iCloud-synced `~/Documents`, so PyInstaller's output carries `com.apple.FinderInfo` + `com.apple.fileprovider.fpfs#P` xattrs that the fileprovider **re-applies in place**, and `codesign` rejects them ("resource fork, Finder information, or similar detritus not allowed"). Fix: the script now **stages the `.app` to a local `$TMPDIR` dir**, `xattr -cr`s it there (where the strip sticks), signs inside-out, then places the result. Applies to the real MAS build too.

**2. App robustness (`mfl_desktop/__main__.py`).** `_open_repository_with_fallback()` logs the resolved DB path and, if a sandbox first-run location can't be opened, **falls back to the container** with a user warning instead of crashing; `_prompt_first_run_location` logs the picked folder.

**3. KEY FINDING — a live SQLite database cannot run on iCloud Drive under the sandbox.** Verified twice from the container log: the first-run picker pointed at the owner's iCloud `…/com~apple~CloudDocs/Banktivity Exports/` folder, and a later File ▸ Open targeted the real `mfl_dev_windows5.mfl` there (14 MB, fully downloaded — *not* evicted); both failed with `sqlite3.OperationalError: unable to open database file`, with **no sandbox-denial logged**. A **local** copy of the very same file opens fine (28 accounts). iCloud paths are fileprovider-managed; SQLite's WAL sidecar creation + POSIX file locking aren't permitted there under the sandbox even with the user-selected-file entitlement, and there is **no entitlement that fixes it** (the iCloud entitlement is for the app's *own* container, not a user's CloudDocs file). This is consistent with the general truth that a live SQLite/WAL DB on any cloud-synced folder risks corruption — the sandbox just turns the silent risk into a hard, early failure.

**Consequence — the file-location sub-decision is RE-OPENED.** The "first-run choose a folder" step (built and working for a *local* folder) is, in practice, a trap for this owner: their `~/Documents`/`~/Desktop` are iCloud-synced and their real `.mfl` lives in iCloud, so the picker leads straight to a location SQLite can't use. The live working file must be **local**. Options now in front of the owner (not yet chosen):
1. **Container by default** (local, WAL-safe, zero friction); iCloud holds only *exported* copies/snapshots; add "Reveal in Finder" + an explicit "import an existing file" flow that copies a chosen file into local storage. *(Recommended.)*
2. **Keep the folder picker but detect & refuse cloud paths**, steering to a local folder.
3. **Container default + optional local move** via Manage Data ▸ Locations (cloud paths blocked).

This **supersedes the original "first-run choose a folder" decision** above for the iCloud case; the picker stays only if paired with cloud-path rejection (option 2/3). **Migration note:** to use existing data under the sandboxed app, the user keeps a **local** working copy and uses iCloud for backups — not the live file. For testing now, the owner's real file was copied into the container (`…/Containers/life.myfinancial.app/Data/Documents/mfl_dev_windows5.mfl`) and opened from there.

**Still pending:** the owner's file-location choice (1/2/3), then wiring it + a clean "open/import an existing file" flow; and the account-gated signing/upload.

## Rejected alternatives (summary)

- **Stay macOS-out-of-scope (ADR-123 as-is):** rejected by the owner's decision to pursue MAS now.
- **Direct Developer-ID DMG instead of MAS:** the ADR-104/K1 scaffold already exists and needs no sandbox, but the owner chose the Store (handles trust, payment, tax, auto-update). Kept in-tree as a dev convenience only.
- **MAS + direct DMG in parallel:** rejected — dual channels double the support/trust/update surface.
- **Container default file:** originally rejected as invisible/bad for a finance app — but **revisited by the 2026-06-30 addendum**: a live SQLite DB can't run on iCloud (where this owner's visible folders point), so a local container default is now the recommended option, paired with "Reveal in Finder" + export-to-iCloud for backups.
- **Free + StoreKit IAP:** rejected — native integration cost for no benefit at a one-time price.
